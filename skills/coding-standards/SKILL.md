# Coding Standards Skill

プロジェクトのコーディング規約と品質ツールを設定・維持するスキル。

**開始時に宣言:** 「coding-standards スキルでコーディング規約を設定します。」

**重要: このスキルは言語・フレームワークに依存しない共通ルールを定義する。** フレームワーク固有の規約は docs/ の該当ファイルを Read して参照すること。

---

## 全体像

```
1. コード規約（共通） — どの言語・フレームワークでも守るべきルール
2. フレームワーク固有の規約 — docs/ から該当するものを Read
3. 自動フォーマット・静的解析 — プロジェクトに合ったツールを設定
4. 品質ゲート — コミット前の自動チェック
```

---

## フレームワーク固有の規約（docs/）

プロジェクトの技術スタックに応じて、以下の docs/ を Read すること:

| プロジェクトの技術スタック | 参照する docs/ |
|---|---|
| TypeScript を使用 | `docs/typescript.md` |
| React / Next.js を使用 | `docs/react-nextjs.md` |

**docs/ にないフレームワーク・言語の場合:**
プロジェクトの既存コードと公式ドキュメントから規約を判断する。

## 1. コード規約（共通）

言語やフレームワークに関わらず適用される原則。

### 命名の原則

```
✅ 名前は「何をするか」「何であるか」が読み取れるものにする
✅ 略語は避ける（usr → user、btn → button）
✅ 一貫した命名パターンを使う（プロジェクト内で統一）
✅ bool 値は is/has/can/should プレフィックスで意図を明確にする
❌ 1文字の変数名（ループカウンタの i, j 等は例外）
❌ 名前に型情報を含める（userArray → users）
```

### 関数・メソッドの原則

```
✅ 1つの関数は1つの責務のみ担う
✅ 引数は少なく保つ（3つ以下を目安。多い場合はオブジェクトにまとめる）
✅ 副作用（外部状態の変更）がある関数とない関数を分離する
✅ 早期リターンでネストを浅く保つ
❌ 1つの関数に複数の責務を詰め込む
❌ 深いネスト（3段以上の if/for のネスト）
```

### エラーハンドリングの原則

```
✅ エラーは適切な粒度でキャッチする（大きすぎる try-catch を避ける）
✅ エラーメッセージには「何が起きたか」「何をすべきか」を含める
✅ エラー時は安全側に倒す（fail-safe: エラー時はアクセス拒否等）
✅ 予期しない入力に対するガードを入れる
❌ エラーを無視する（空の catch ブロック）
❌ ユーザーに内部エラーの詳細を見せる
```

### コメントの原則

```
✅ 「なぜ（WHY）」をコメントに書く
✅ 複雑なロジックの意図を説明する
❌ 「何をしているか（WHAT）」をコメントに書く（コードを読めばわかる）
❌ コメントアウトしたコードを残す（Git で履歴管理する）
```

### ファイル構成の原則

```
✅ 1ファイルにつき1つの主要な責務
✅ ファイル名は内容を反映する
✅ 関連するコードは近くに置く（コロケーション）
✅ ディレクトリ構成は project-structure スキルに従う
```

### DB 操作の原則

```
✅ ORM やクエリビルダーを優先する。生クエリは原則使わない
✅ DB 操作はデータアクセス層に集約する
✅ トランザクションが必要な操作は明示する
✅ N+1 問題に注意する
❌ ユーザー入力をクエリに直接組み込む（インジェクションの危険）
```

### API 設計の原則

```
✅ レスポンス形式を統一する（成功 / エラーの型を揃える）
✅ HTTP ステータスコードを正しく使う
✅ 入力値はスキーマでバリデーションする
✅ エラーレスポンスにはコードとメッセージを含める
❌ エラー時に 200 を返す
❌ 入力をバリデーションせずに処理する
```

## 2. 自動フォーマット・静的解析

**プロジェクトの言語・フレームワークに合ったツールを設定する。** 以下は共通の考え方:

### ツール選定の原則

| 役割 | やること | 例 |
|---|---|---|
| フォーマッター | コードの見た目を統一する（インデント、改行等） | Prettier, Black, gofmt |
| 静的解析（linter） | コードの問題を検出する（未使用変数、潜在バグ等） | ESLint, Ruff, golangci-lint |
| 型チェッカー | 型の整合性を検証する | tsc, mypy, go vet |

### 設定ファイルの原則

```
✅ 設定ファイルはリポジトリにコミットする（チーム全員が同じルール）
✅ フォーマッターと linter の競合を避ける設定にする
✅ 厳格なルールを基本にし、必要に応じて緩める
✅ IDE の設定も揃える（.editorconfig 等）
❌ 各開発者がローカルで設定を変える
```

### フレームワーク固有の設定

フレームワーク固有のツール設定テンプレートは docs/ を参照:

| 技術スタック | 参照先 |
|---|---|
| TypeScript + React/Next.js | `docs/react-nextjs.md` のツール設定セクション |

## 3. 品質ゲート（コミット前の自動チェック）

### 仕組みの概要

```
開発者がコミットを実行
  ↓
pre-commit フックが発動
  ↓
変更されたファイルのみにチェックを実行:
  - フォーマッター（自動修正）
  - linter（自動修正可能なものは修正）
  ↓
commit-msg フックが発動
  ↓
コミットメッセージを検証:
  - git-conventions スキルの Conventional Commits 形式に従っているか
  ↓
すべてパス → コミット成功
いずれか失敗 → コミット拒否（修正してから再コミット）
```

### ツール

品質ゲートの実現に使うツールはエコシステムによって異なる:

| エコシステム | pre-commit ツール | commit-msg 検証 |
|---|---|---|
| Node.js | Husky + lint-staged | commitlint |
| Python | pre-commit framework | commitizen |
| Go | pre-commit framework | commitlint (npx) |
| 汎用 | lefthook, pre-commit | — |

**具体的なインストール手順と設定テンプレートは docs/ を参照。**

## 既存プロジェクトへの適用手順

```
Step 1: 現状確認
│  既存のフォーマッター / linter 設定を確認
│  プロジェクトの技術スタックを確認
│
Step 2: docs/ から該当するテンプレートを参照
│  技術スタックに合った docs/ ファイルを Read
│
Step 3: ツールのインストール
│  不足しているツールをインストール
│  設定ファイルを作成（既存設定がある場合はマージ）
│
Step 4: 初回フォーマット
│  全ファイルをフォーマット
│  自動修正可能な lint エラーを修正
│  ※ この変更は「chore: apply coding standards」として1コミットにまとめる
│
Step 5: 品質ゲートの設定
│  pre-commit フック + コミットメッセージ検証を設定
│
Step 6: チームへの共有
│  設定ファイルをコミット
│  README または docs/ に設定の説明を追記
```

---

## Red Flags

以下を検出したら **即座に指摘** すること:

1. ❌ フォーマッター / linter の設定ファイルがない
2. ❌ コミット前の自動チェックが未設定
3. ❌ エラーを無視する空の catch ブロック
4. ❌ デバッグ用のログ出力が残っている
5. ❌ コメントアウトしたコードが残っている
6. ❌ ユーザー入力をバリデーションせずに使っている
7. ❌ 生クエリにユーザー入力を直接組み込んでいる
8. ❌ 1つの関数に複数の責務が詰め込まれている
