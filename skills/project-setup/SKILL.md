# Project Setup Skill

既存プロジェクトの開発環境を構築するスキル。

**開始時に宣言:** 「project-setup スキルで環境構築を行います。」

**重要: このスキルは特定のフレームワークやツールを前提にしない。** 具体的なコマンドや設定はプロジェクトの README、package.json、設定ファイルから読み取ること。

---

## 全体フロー

```
Phase 1: プロジェクトの把握（何が使われているか調べる）
Phase 2: ランタイム・依存の準備
Phase 3: 環境変数・シークレットの設定
Phase 4: データストア・外部サービスの起動
Phase 5: 品質ツールの確認
Phase 6: 動作確認
Phase 7: CLAUDE.md の確認・生成
```

**原則: README にセットアップ手順が書かれている場合は、それを最優先で参照する。** このスキルは README に書かれていない観点を補完する。

---

## Phase 1: プロジェクトの把握

**目的:** このプロジェクトが何で構成されているかを理解する。

以下のファイルを確認し、プロジェクトの全体像を把握する:

| 確認するもの | 何がわかるか |
|---|---|
| README.md | セットアップ手順、プロジェクト概要 |
| package.json（または同等の依存定義ファイル） | 使っているライブラリ、スクリプト一覧 |
| ロックファイルの種類 | パッケージマネージャの判定 |
| ランタイムバージョン指定ファイル（.nvmrc, .python-version, .tool-versions 等） | 必要なランタイムバージョン |
| .env.example / .env.sample / .env.template | 必要な環境変数の一覧 |
| docker-compose.yml / Dockerfile | コンテナ利用の有無 |
| 設定ファイル一覧（lint、formatter、型チェック等） | 品質ツールの導入状況 |
| CLAUDE.md | 既存のプロジェクト設定 |

**Phase 1 チェックリスト:**

- [ ] README を読んだか
- [ ] 使用言語・フレームワークを把握したか
- [ ] パッケージマネージャを特定したか
- [ ] 必要なランタイムバージョンを確認したか
- [ ] DB や外部サービスの利用有無を確認したか
- [ ] 環境変数のテンプレートがあるか確認したか
- [ ] README にセットアップ手順が書かれているか確認したか

---

## Phase 2: ランタイム・依存の準備

**目的:** プロジェクトが動くために必要なソフトウェアと依存パッケージを揃える。

### Step 2-1: ランタイムのバージョン合わせ

- [ ] プロジェクトが要求するランタイムバージョンを確認した
- [ ] そのバージョンがローカル環境にインストールされている
- [ ] バージョン管理ツール（nvm, pyenv, asdf, mise 等）で切り替え済み

**バージョンが合っていないと、後の工程で原因不明のエラーが出る。ここは省略しない。**

### Step 2-2: 依存パッケージのインストール

- [ ] ロックファイルに忠実なインストールコマンドを使った（ロックファイルを書き換えるコマンドではなく、再現性のあるコマンドを使う）
- [ ] インストールがエラーなく完了した
- [ ] postinstall 等の自動実行スクリプトが正常に完了した

**判断基準: ロックファイルが存在する場合は、それに完全一致する方法でインストールする。** ロックファイルを更新するインストールはチーム開発で問題を起こす。

---

## Phase 3: 環境変数・シークレットの設定

**目的:** アプリが外部サービスや設定値を読み込めるようにする。

### Step 3-1: テンプレートからのコピー

- [ ] 環境変数のテンプレートファイル（.env.example 等）があるか確認した
- [ ] テンプレートを開発用の環境変数ファイルにコピーした

### Step 3-2: 値の設定

- [ ] 各環境変数の意味を把握した（README やテンプレート内のコメントを参照）
- [ ] DB接続情報を設定した（DB を使う場合）
- [ ] 認証関連のシークレットを設定した（認証を使う場合）
- [ ] 外部 API のキーを設定した（外部サービスを使う場合）
- [ ] ローカル開発用の URL を設定した

### Step 3-3: 安全性確認

- [ ] シークレットを含むファイルが .gitignore に含まれているか確認した
- [ ] 本番環境のシークレットをローカルに使っていないか確認した
- [ ] 「クライアントに公開される環境変数」と「サーバーのみの環境変数」の区別を理解した

**環境変数がわからない場合:** コード内で環境変数を参照している箇所を検索して、必要な変数の一覧を洗い出す。

---

## Phase 4: データストア・外部サービスの起動

**目的:** アプリが依存する外部システムを起動し、接続できる状態にする。

**Phase 1 で「DB や外部サービスなし」と判明した場合は、このフェーズをスキップ。**

- [ ] DB が必要か確認した
- [ ] DB の起動方法を確認した（Docker、ローカルインストール、クラウド等）
- [ ] DB を起動した
- [ ] マイグレーション（DBスキーマの適用）が必要か確認した
- [ ] マイグレーションを実行した
- [ ] 初期データ（シード）の投入が必要か確認した
- [ ] シードデータを投入した
- [ ] DB に接続できることを確認した
- [ ] その他の外部サービス（Redis、メールサーバー等）が必要か確認し、起動した

---

## Phase 5: 品質ツールの確認

**目的:** コード品質を維持するツールが揃っているか確認する。

| 確認項目 | 確認方法 | ない場合の対応 |
|---|---|---|
| 静的解析（linter） | 設定ファイルの有無 | coding-standards スキルで設定 |
| フォーマッター | 設定ファイルの有無 | coding-standards スキルで設定 |
| 型チェック | 型定義ファイル・設定の有無 | プロジェクトの言語に応じて設定 |
| コミット前チェック | Git hook の設定有無 | coding-standards スキルで設定 |
| コミットメッセージ検証 | commitlint 等の設定有無 | coding-standards スキルで設定 |

- [ ] 静的解析ツールが設定されているか
- [ ] フォーマッターが設定されているか
- [ ] 型チェックが設定されているか（型付き言語の場合）
- [ ] コミット前の自動チェックが設定されているか

**不足しているツールがある場合** → coding-standards スキルに従ってインストール・設定する。

---

## Phase 6: 動作確認

**目的:** すべてが正しく設定されたことを検証する。

以下を「上から順番に」確認する。上が失敗していると下も失敗する可能性が高いため。

### 確認順序:

```
1. 型チェック — コードが文法的に正しいか
2. 静的解析（lint） — コードスタイルの問題がないか
3. テスト — 既存テストがすべて通るか
4. ビルド — 本番用ビルドが成功するか
5. 開発サーバー — ローカルで起動し画面が表示されるか
```

**Phase 6 チェックリスト:**

- [ ] 型チェックがパスする（型付き言語の場合）
- [ ] lint がパスする
- [ ] テストがパスする（テストが存在する場合）
- [ ] ビルドが成功する
- [ ] 開発サーバーが起動し、ブラウザ等で動作確認できる

**いずれかが失敗する場合:**
- まず Phase 1〜4 の設定漏れがないか見直す
- README のトラブルシューティングを確認する
- それでも解決しない場合は systematic-debugging スキルで原因を特定する

---

## Phase 7: CLAUDE.md の確認・生成

**目的:** AI がプロジェクトを正しく理解できる状態にする。

- [ ] CLAUDE.md が存在するか確認
- [ ] 存在しない場合 → **claude-md-generator** スキルで生成
- [ ] 存在するが情報が古い場合 → **claude-md-generator** スキルで更新
- [ ] CLAUDE.md の内容が現在のプロジェクト状態と一致しているか確認

---

## 環境構築の完了条件

以下がすべて満たされたら、環境構築は完了:

```
✅ 必要なランタイムが正しいバージョンでインストールされている
✅ 依存パッケージがインストールされている
✅ 環境変数が設定されている
✅ データストア・外部サービスが起動し接続できる（必要な場合）
✅ 型チェックがパスする（型付き言語の場合）
✅ lint がパスする
✅ テストがパスする（テストがある場合）
✅ ビルドが成功する
✅ 開発サーバーが起動し動作確認できる
✅ 品質ツールが設定されている
✅ CLAUDE.md が存在し最新状態である
```

---

## Red Flags

以下を検出したら **即座に指摘** すること:

1. ❌ README を読まずにセットアップを始めた
2. ❌ ランタイムバージョンを確認せずに依存をインストールした
3. ❌ ロックファイルを書き換えるインストールコマンドを使った
4. ❌ シークレットを含むファイルをコミットしようとしている
5. ❌ 本番環境のシークレットをローカルで使っている
6. ❌ テストが失敗している状態で開発を始めようとしている
7. ❌ CLAUDE.md なしで作業を始めようとしている
8. ❌ Phase の順番を飛ばしている（前の Phase が完了していない状態で先に進む）
