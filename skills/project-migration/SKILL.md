# 既存プロジェクトへの claude-code-harness 導入

## 概要

既に開発中のプロジェクトに claude-code-harness を途中から導入するためのスキル。
既存のコード・ドキュメント・テストから、architecture.md と feature ファイル群を一括生成する。

**基本原則：**
- 既存ドキュメントは削除しない。`docs/legacy/` に保管し、feature ファイルから参照する
- 導入作業自体を feature ファイル（F-000）のタスクとして管理し、Ralph Loop で自動化する
- 一度に完璧を目指さない。まず全体像を作り、詳細は後から精緻化する

## 前提条件

- プロジェクトに既存のコードがある
- 既存のドキュメント（要件定義書、設計書、README 等）がある程度存在する
- claude-code-harness プラグインがインストール済み

## 全体の流れ

```
Phase 1: 現状調査（コード・ドキュメント・テストを調査し、機能一覧を作る）
    ↓
Phase 2: 導入準備（docs/ 構造の作成、既存ドキュメントの保管、architecture.md の生成）
    ↓
Phase 3: feature ファイルの一括生成（機能一覧をもとに Ralph Loop で自動生成）
    ↓
Phase 4: 検証・仕上げ（整合性チェック、CLAUDE.md 生成）
```

Phase 1〜2 はこのスキルの対話で行う。Phase 3〜4 は Ralph Loop で自動実行する。

---

## Phase 1: 現状調査

### Step 1-1: コード構造の調査

プロジェクトのディレクトリ構造、技術スタック、主要なモジュールを把握する。

**調査項目:**
- ディレクトリ構成（src/ の中身、主要なディレクトリの役割）
- 技術スタック（言語、フレームワーク、DB、外部サービス）
- エントリポイント（API ルート、ページ、CLI コマンド等）
- 共通処理（認証、バリデーション、エラーハンドリング等）

**成果物:** `docs/notes/{日付}_migration-code-survey.md`

### Step 1-2: 既存ドキュメントの調査

既存のドキュメントがどこに何があるかを把握する。

**調査項目:**
- ドキュメントの場所（README、Wiki、docs/ 内ファイル、Issue、外部ツール等）
- 各ドキュメントの内容と鮮度（最終更新日、コードとの乖離があるか）
- ドキュメント間の関係（どれが最新の正なのか）

**成果物:** `docs/notes/{日付}_migration-doc-survey.md`

### Step 1-3: 既存テストの調査

テストの現状を把握する。

**調査項目:**
- テストの場所とフレームワーク
- テストの種類（単体テスト、統合テスト、E2E テスト）
- カバレッジの概況（テストがある領域、ない領域）
- テストが検証している内容（どの機能・仕様に対応するか）

**成果物:** `docs/notes/{日付}_migration-test-survey.md`

### Step 1-4: 機能一覧の作成

調査結果から、プロジェクトの機能を洗い出してリスト化する。

**機能の粒度の目安:**
- 「ユーザー登録」「商品検索」「決済」のように、ユーザーから見て独立した機能を1つの feature とする
- 「認証」「通知」のように横断的な機能も1つの feature とする
- 迷ったら大きめに切る。後から分割はできるが、統合は面倒

**機能一覧のフォーマット:**

```markdown
# 機能一覧

| F番号 | 機能名 | 概要 | 既存ドキュメント | 既存テスト | 優先度 |
|-------|-------|------|-----------------|-----------|--------|
| F-001 | ユーザー登録 | メールアドレスでの登録・ログイン | 要件定義書 3.1 | auth.test.ts | 高 |
| F-002 | 商品検索 | キーワード・カテゴリによる検索 | 設計書 4.2 | search.spec.ts | 高 |
| F-003 | 決済 | クレジットカード決済 | なし | なし | 中 |
...
```

**ユーザーとの確認:**
機能一覧を作成したら、必ずユーザーに確認する。以下を聞く：
- 抜けている機能はないか
- 粒度は適切か（大きすぎ / 小さすぎ）
- 優先度は合っているか
- F番号の割り当ては良いか

**成果物:** `docs/notes/{日付}_migration-feature-list.md`

## Phase 2: 導入準備

### Step 2-1: docs/ ディレクトリ構造の作成

```
docs/
├── architecture.md
├── features/
├── legacy/                    ← 既存ドキュメントの保管場所
│   └── （元のディレクトリ構造を保持）
└── notes/
    ├── {日付}_migration-code-survey.md
    ├── {日付}_migration-doc-survey.md
    ├── {日付}_migration-test-survey.md
    └── {日付}_migration-feature-list.md
```

### Step 2-2: 既存ドキュメントの保管

既存ドキュメントを `docs/legacy/` にコピーする。

**ルール:**
- **コピーする。移動ではない。** 元のファイルは元の場所に残す
- 元のディレクトリ構造をそのまま保持する
- コピー元とコピー先の対応を `docs/legacy/README.md` に記録する

**docs/legacy/README.md のフォーマット:**

```markdown
# 既存ドキュメント（導入時保管）

移行日: {日付}

このディレクトリには、claude-code-harness 導入時点の既存ドキュメントを保管している。
feature ファイルの各項目に「出典」として参照される。

## ファイル一覧

| ファイル | 元の場所 | 内容 |
|---------|---------|------|
| 要件定義書.md | /docs/requirements.md | ユーザー要件の定義 |
| 設計書/ | /docs/design/ | 各機能の設計ドキュメント |
| API仕様.yaml | /api/openapi.yaml | OpenAPI 仕様 |
```

**元のファイルの扱い:**
ユーザーに確認する。選択肢は：
- 元のファイルもそのまま残す（両方に存在する状態）
- 元のファイルは削除して `docs/legacy/` に一本化する
- 元のファイルに「`docs/legacy/` に移管済み」と注記を入れる

### Step 2-3: architecture.md の生成

Phase 1 の調査結果と既存コードから、architecture.md を生成する。

**手順:**
1. document-lifecycle スキルの architecture.md テンプレートに従う
2. 既存コードの DB スキーマ → テーブル一覧
3. 既存コードの API ルート → エンドポイント一覧
4. 既存コードの共通処理 → 共通ルール
5. 技術スタックの概要

**注意:** 完璧を目指さない。Phase 3 で feature ファイルを作る過程で追記される情報も多い。

## Phase 3: feature ファイルの一括生成

### Step 3-1: 導入用 feature ファイル（F-000）の生成

Phase 1 で作成した機能一覧をもとに、導入作業全体を管理する feature ファイルを生成する。

**F-000 テンプレート:**

```markdown
---
id: F-000
feature: claude-code-harness 導入
status: in-progress
progress: 0/{タスク総数}
---

## 要求

- F-000-R01: 既存プロジェクトの全機能が feature ファイルとして記録されている
- F-000-R02: architecture.md にプロジェクト全体の設計情報がまとまっている
- F-000-R03: CLAUDE.md が生成されている
- F-000-R04: 既存ドキュメントが docs/legacy/ に保管されている

## 要件

- F-000-S01: 各 feature ファイルに要求・要件・技術仕様が記載されている （参照: R01）
- F-000-S02: 既存テストと要求・要件の対応が記録されている （参照: R01）
- F-000-S03: architecture.md が document-lifecycle のテンプレートに準拠している （参照: R02）
- F-000-S04: 既存ドキュメントの出典が feature ファイルに記録されている （参照: R04）

## タスク

<!-- Phase 3: feature ファイル生成（機能一覧から自動展開） -->
- [ ] F-000-T01: F-001_{機能名}.md を生成する （出典: docs/legacy/xxx）
- [ ] F-000-T02: F-002_{機能名}.md を生成する （出典: docs/legacy/xxx）
- [ ] F-000-T03: F-003_{機能名}.md を生成する （出典: docs/legacy/xxx）
...（機能一覧の数だけ続く）

<!-- Phase 4: 検証・仕上げ -->
- [ ] F-000-T{N-2}: architecture.md を全 feature ファイルの情報で補完する
- [ ] F-000-T{N-1}: CLAUDE.md を生成する
- [ ] F-000-T{N}: 全 feature ファイルと architecture.md の整合性を検証する
```

**タスクの自動展開:**
機能一覧の各行を1つのタスクに変換する。各タスクの内容は「既存コードと既存ドキュメントを読んで、feature ファイルを生成する」。

### Step 3-2: 各 feature ファイルの生成（Ralph Loop が実行）

Ralph Loop が F-000 のタスクを上から順に処理する。各タスクで以下を行う：

```
ralph --todo docs/features/F-000_harness-migration.md
```

**各タスクで AI が行う作業:**

1. 機能一覧から対象機能の情報を読む
2. 既存ドキュメント（docs/legacy/）から関連する記述を探す
3. 既存コードから技術仕様（API/DB/UI）を読み取る
4. 既存テストから、何が検証されているかを把握する
5. document-lifecycle の feature テンプレートに従って feature ファイルを生成する
6. architecture.md の一覧に新しい API/DB を追記する

**feature ファイルへの出典記録:**

既存ドキュメントから内容を抽出した場合、出典を残す。

```markdown
## 要求

- F-001-R01: ユーザーがメールアドレスで登録できる
  （出典: docs/legacy/要件定義書.md > 3.1 ユーザー登録）

## 要件

- F-001-S01: メールアドレスの形式バリデーションを行う （参照: R01）
  （出典: docs/legacy/設計書/認証.md > バリデーション仕様）
```

**既存テストの紐づけ:**

既存テストがどの要求・要件を検証しているかを、feature ファイルの未決事項または notes に記録する。

```markdown
## 未決事項

- 既存テスト auth.test.ts は R01, S01 に対応すると推定。要確認。
```

**タスクセクションについて:**
導入時はタスクセクションを空にする。タスクは新しい開発や修正が発生したときに追加する。既に動いている機能に対して改めて「実装する」タスクは不要。

```markdown
## タスク

（既存機能のため、現時点ではタスクなし。新規開発・修正時に追加する。）
```

**frontmatter の status:**
既に動いている機能なので `done` にする。タスクが追加されたら `in-progress` に変わる。

```markdown
---
id: F-001
feature: ユーザー登録
status: done
progress: 0/0
---
```

## Phase 4: 検証・仕上げ

### Step 4-1: architecture.md の補完

Phase 3 で feature ファイルを作る過程で蓄積された情報を architecture.md に反映する。

**確認項目:**
- 全 feature ファイルの API が architecture.md のエンドポイント一覧に含まれているか
- 全 feature ファイルの DB が architecture.md のテーブル一覧に含まれているか
- 共通ルール（認証方式、レスポンス形式等）が記載されているか

### Step 4-2: CLAUDE.md の生成

claude-md-generator スキルを使って CLAUDE.md を生成する。

### Step 4-3: 整合性の検証

**チェック項目:**
- 機能一覧の全機能に feature ファイルが存在するか
- F番号に欠番がないか（あっても問題ないが、意図的かどうか確認）
- feature ファイルの管理番号（R, S, API, DB, UI）が重複していないか
- architecture.md と feature ファイルの技術仕様に矛盾がないか
- docs/legacy/ の全ファイルが docs/legacy/README.md に記録されているか

## PROMPT.md テンプレート（Ralph Loop 用）

Phase 3〜4 を Ralph Loop で自動実行するための PROMPT.md。F-000 と一緒にプロジェクトルートに配置する。

```markdown
あなたは自律的に作業するエージェントです。

# 参照ファイル

- **進捗ファイル**: `./docs/features/F-000_harness-migration.md`
- **この指示書**: `./PROMPT.md`
- **作業ログ**: `./SCRATCH.md`

# 手順

1. 進捗ファイルを読んでください
2. 未完了 ([ ]) のタスクを1つだけ選んでください
3. そのタスクを以下のルールに従って実行してください
4. 完了したら進捗ファイルのそのタスクを [x] に変えてください
5. 何をやったかを `./SCRATCH.md` に1〜2行で追記してください

# feature ファイル生成のルール

- document-lifecycle スキルの feature テンプレートに従う
- 既存コード（src/ 等）を読んで技術仕様（API/DB/UI）を抽出する
- 既存ドキュメント（docs/legacy/）を読んで要求・要件を抽出する
- 既存ドキュメントから内容を抽出した場合、出典を記録する
  例: （出典: docs/legacy/要件定義書.md > 3.1 ユーザー登録）
- 既存テストがあれば、どの要求・要件に対応するか推定して未決事項に記録する
- タスクセクションは空にする（既存機能のため）
- frontmatter の status は done にする
- architecture.md のAPI一覧・テーブル一覧に追記する
- 不明点は `./SCRATCH.md` に「QUESTION:」として記録してスキップ

# 制約

- 1回の実行で扱うタスクは1つだけ
- 進捗ファイルに書かれていないことはやらない
- 既存のコードやテストを変更しない
- 推測で補った部分は必ず「推定」と明記する
```

## Red Flags — 以下に該当したら立ち止まること

| Red Flag | 正しい対応 |
|----------|-----------|
| 「既存ドキュメントを削除した」 | docs/legacy/ にコピーする。元のファイルの扱いはユーザーに確認する |
| 「既存コードを変更した」 | 導入フェーズではコードを一切変更しない。ドキュメント生成のみ |
| 「機能一覧をユーザーに確認せずに feature ファイルを作り始めた」 | Phase 1-4 の確認を必ず経ること |
| 「出典を書かなかった」 | 既存ドキュメントから抽出した内容には必ず出典を残す |
| 「完璧な feature ファイルを作ろうとして進まない」 | 不明点は未決事項に書いて先に進む。後から精緻化する |
| 「既存テストの対応づけを確定した」 | 「推定」と明記する。確定はユーザーが行う |
| 「タスクセクションに実装タスクを書いた」 | 既に動いている機能にタスクは不要。空にする |
