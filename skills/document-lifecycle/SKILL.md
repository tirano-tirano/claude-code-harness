# ドキュメント駆動開発（Document-Driven Development）

## 概要

コードを書く前にドキュメントを書く。コードを変えたらドキュメントも更新する。

**基本原則：** ドキュメントが存在しない機能は、合意されていない機能である。

## ドキュメントの分類

「AIが実装中に開くファイルか？」で2つに分ける。

- **性質A（実装時に参照する）** → 独立ファイルとして管理（architecture.md, features/）
- **性質B（後で振り返る記録）** → docs/notes/ に集約（ブレスト、ADR、日誌、セキュリティレビュー等）

## docs/ ディレクトリ構造

```
docs/
├── architecture.md                          # 全体設計（DB一覧、API一覧、共通方針）
├── features/                                # 機能ごとの全情報（1機能 = 1ファイル）
│   ├── F-001_user-registration.md
│   ├── F-002_payment.md
│   └── F-003_notification.md
├── legacy/                                  # 既存ドキュメント保管（途中導入時、project-migration で作成）
│   └── （元のディレクトリ構造を保持）
└── notes/                                   # 記録系（ブレスト、ADR、日誌、セキュリティ等）
    ├── 2026-04-17_brainstorm_テーマ.md
    ├── 2026-04-17_adr-001_認証方式.md
    ├── 2026-04-18_security-001_XSS.md
    └── 2026-04-18_journal.md
```

## 各ファイルの役割

### architecture.md — 全体設計

プロジェクト横断の設計情報を1ファイルにまとめる。新機能設計時や横断的な確認時に読む。

含める内容：
- DB全体のテーブル一覧・ER図
- API全エンドポイント一覧
- 共通ルール（認証方式、レスポンス形式、エラーコード体系）
- 技術スタックの概要

### features/{番号}_{機能名}.md — 機能仕様

**1機能に関する全情報を1ファイルにまとめる。** AIはこのファイルだけ読めば実装に入れる。

含める内容：要求、要件、技術仕様（API/DB/UI）、タスクリスト＋進捗チェックボックス

### notes/ — 記録系

時系列の記録。命名規則：`{日付}_{種類}_{テーマ}.md`

種類の例：brainstorm, adr, journal, security

## 管理番号

ドキュメントの追跡・参照を容易にするため、以下の番号体系を使う。

### ファイルレベルの管理番号

| 種類 | プレフィックス | 形式 | 例 |
|------|-------------|------|-----|
| feature ファイル | F | F-001, F-002... | F-001 |
| ADR | ADR | ADR-001, ADR-002... | ADR-003 |
| セキュリティレビュー | SEC | SEC-001, SEC-002... | SEC-001 |
| ブレスト・日誌 | なし | 番号不要（日付とテーマで識別） | — |

### feature ファイル内の管理番号（階層方式）

feature ファイルの中の項目にも番号を振る。全ての種類が独立した通し番号を持ち、関連する項目を「参照」で紐づける（参照方式）。

| 種類 | 形式 | 例 |
|------|------|-----|
| 要求 | {F番号}-R{連番} | F-001-R01, F-001-R02 |
| 要件 | {F番号}-S{連番} | F-001-S01, F-001-S02 |
| API | {F番号}-API-{連番} | F-001-API-01 |
| DB | {F番号}-DB-{連番} | F-001-DB-01 |
| UI | {F番号}-UI-{連番} | F-001-UI-01 |
| タスク | {F番号}-T{連番} | F-001-T01 |

全ての紐づけは「参照」で表現する。1つの項目が複数の親を持てる。

全体像:
```
F-001（feature）
│
├── 要求
│   F-001-R01  ...
│   F-001-R02  ...
│
├── 要件（独立番号 + 要求への参照）
│   F-001-S01  ...  参照: R01
│   F-001-S02  ...  参照: R01
│   F-001-S03  ...  参照: R01, R02（複数の要求にまたがる）
│
├── 技術仕様（独立番号 + 要件への参照）
│   F-001-API-01  ...  参照: S01, S02
│   F-001-DB-01   ...  参照: S01, S03
│   F-001-UI-01   ...  参照: S01, S02
│
└── タスク（独立番号 + 技術仕様への参照）
    F-001-T01  ...  参照: DB-01
    F-001-T02  ...  参照: API-01
    F-001-T03  ...  参照: UI-01
```

### 管理番号の共通ルール

- ファイルレベルの番号は **プロジェクト全体で通し番号** にする（リセットしない）
- ファイル内の番号は **各 feature 内で通し番号** にする
- ファイルレベルの番号は **YAML frontmatter の id フィールド** に記載する
- 新しい番号を振るときは、既存の番号を確認して最大番号の次を使う
- 削除・欠番が生じても番号を再利用しない
- 他のドキュメントから参照するときは管理番号で参照する（例：「F-001-R01-02 を参照」「ADR-003 で決定済み」）
- feature 内での参照は feature 番号を省略してよい（例：タスク内で「参照: API-01」）

### ファイル名と管理番号の対応

```
docs/features/F-001_user-registration.md    ← frontmatter に id: F-001
docs/features/F-002_payment.md              ← frontmatter に id: F-002
docs/notes/2026-04-17_adr-003_認証方式.md    ← frontmatter に id: ADR-003
docs/notes/2026-04-18_security-001_XSS.md   ← frontmatter に id: SEC-001
```

## 出力ドキュメントの書式ルール

- **絵文字を使わない**: 出力するドキュメント（feature ファイル、ADR、セキュリティレビュー、ブレストメモ等）に絵文字を含めない。チェックボックスは `- [ ]` / `- [x]`、ステータスはテキスト（「未修正」「修正済み」等）で表記する
- **Markdown の標準記法を使う**: 見出し（#）、リスト（-）、チェックボックス（- [ ]）、テーブル（|）のみで構成する

## architecture.md と feature ファイルの役割分担

- **architecture.md**: 一覧と共通ルール（横断的な情報）
- **feature ファイル**: 個別の詳細（機能固有の情報）
- 同じ情報の重複を最小限にする
- feature ファイルで新しいAPIやテーブルを定義したら、architecture.md の一覧にも追加する

## feature ファイル テンプレート

```markdown
---
id: F-001
feature: 機能名
status: draft
progress: 0/0
---

## 要求

- F-001-R01: {ユーザー視点の目的・ストーリー1}
- F-001-R02: {ユーザー視点の目的・ストーリー2}

## 要件

- F-001-S01: {具体的条件1} （参照: R01）
- F-001-S02: {具体的条件2} （参照: R01）
- F-001-S03: {具体的条件3} （参照: R01, R02）

## 技術仕様

### API

| ID | メソッド | パス | 概要 | 参照 |
|----|---------|------|------|------|
| F-001-API-01 | POST | /api/xxx | ... | S01, S02 |
| F-001-API-02 | GET | /api/xxx | ... | S03 |

### DB

| ID | テーブル名 | 概要 | 参照 |
|----|-----------|------|------|
| F-001-DB-01 | xxx | ... | S01, S03 |

### UI

| ID | 画面名 | 概要 | 参照 |
|----|-------|------|------|
| F-001-UI-01 | xxx画面 | ... | S01, S02 |

## タスク

受け入れテスト → 統合テスト → 単体テスト → 実装の順で並べる（外側→内側のテスト駆動）。
テストは書いた時点では失敗してよい（Red 状態）。実装タスクで全テストを通す。
Ralph Loop は見出しを無視して `- [ ] F-xxx-Txx` 行を上から順に1つずつ処理する。

<!-- 受け入れテスト（要求から導出） -->
- [ ] F-001-T01: ユーザーが検索できる E2E テストを書く （参照: R01）
<!-- 統合テスト（要件から導出） -->
- [ ] F-001-T02: 検索API + DB の統合テストを書く （参照: S01, API-01, DB-01）
- [ ] F-001-T03: 検索画面 + API の統合テストを書く （参照: S02, UI-01, API-01）
<!-- 単体テスト + 実装（技術仕様から導出、DB → API → UI） -->
- [ ] F-001-T04: xxx テーブルの単体テストを書く （参照: DB-01）
- [ ] F-001-T05: xxx テーブルのマイグレーションを実装する （参照: DB-01）
- [ ] F-001-T06: 検索APIの単体テストを書く （参照: API-01）
- [ ] F-001-T07: 検索APIを実装する （参照: API-01）
- [ ] F-001-T08: xxx画面の単体テストを書く （参照: UI-01）
- [ ] F-001-T09: xxx画面を実装する （参照: UI-01）

## 未決事項

- {まだ決まっていないこと}

## 変更履歴

開発途中で要求・要件・技術仕様を変更・具体化した場合に記録する（詳細は development-cycle の共通ルール5 参照）。

<!-- 形式: {日付} {対象}: {変更内容と理由} -->
```

### タスクの書き方（外側→内側のテスト駆動）

タスクセクションは「テスト → 実装」で構成し、ドキュメントの各レベルから対応するテスト種別を導出する。

**ドキュメントとテスト種別の対応:**

| ドキュメントの層 | テスト種別 | 検証する内容 |
|----------------|----------|------------|
| 要求（R01, R02...） | 受け入れテスト | ユーザー視点のストーリーが達成されるか |
| 要件（S01, S02...） | 統合テスト | 複数コンポーネントを跨いで条件を満たすか |
| 技術仕様（API/DB/UI） | 単体テスト | 個別コンポーネントが仕様通りに動くか |

**タスクの順序（外側→内側）:**

```
1. 受け入れテスト（ユーザーストーリーの E2E テストでゴールを定義）
2. 統合テスト（コンポーネント間の連携テストで中間ゴールを定義）
3. 単体テスト + 実装（DB → API → UI の順。テストと実装をペアで進める）
```

まず受け入れテストと統合テストを書いて「何を満たせば完成か」を先に定義する。その後、単体テスト → 実装のペアで部品を作っていく。全部品が揃えば、先に書いた統合テスト・受け入れテストも通るようになる。

Ralph Loop はタスクを上から順に1つずつ処理するため、この順序で並べることで外側→内側のテスト駆動開発が自然に実現される。

**テストは書いた時点では失敗してよい（Red 状態）。** 受け入れテストや統合テストを書いた段階では実装がまだないので、テストが失敗するのは正常。Ralph Loop の各イテレーションでは「テストを書いた」こと自体がタスク完了の条件であり、テストが通ることは条件ではない。

**テストタスクの参照ルール:**
- 受け入れテスト: 検証対象の要求を参照する
- 統合テスト: 検証対象の要件と、関連する技術仕様を参照する
- 単体テスト: 検証対象の技術仕様を参照する

**テストタスクがない場合:**
テストが不要な作業（設定ファイルの追加、環境構築など）はテストタスクなしの単独タスクでよい。全タスクにテストを強制するものではない。

### バグ修正タスクの追記ルール

バグが見つかったら、そのバグが属する feature ファイルのタスクセクションに修正タスクを追記する。

**バグ修正タスクのテンプレート:**

```markdown
<!-- バグ修正: {バグの概要} ({日付}) -->
- [ ] F-001-T10: {バグの再現テストを書く} （参照: {関連する技術仕様}）
- [ ] F-001-T11: {修正内容} （参照: {関連する技術仕様}）
```

**バグ修正タスクの流れ（テスト駆動）:**

```
1. 再現テストを書く（バグが起きることを確認するテスト → 失敗する状態を確認）
2. 修正する（再現テストが成功するようにコードを変更）
```

再現テストを先に書く理由は3つある:
- 修正が正しいか機械的に確認できる
- 将来同じバグが再発したときに自動的に検出できる（回帰テスト）
- バグの原因を正確に理解してから修正に入れる

**タスク番号の振り方:**
- 既存タスクの最大番号の次を使う（例: T09 まで存在するなら T10 から）
- 番号は欠番が生じても再利用しない

**frontmatter の更新:**
- `status`: `done` → `in-progress` に戻す
- `progress`: タスク総数を更新する（例: `9/9` → `9/11`）
- 修正完了後: 全タスクが `[x]` になったら `status` を `done` に戻す

**HTMLコメントで日付とバグ概要を残す:**
既存の完了済みタスクと区別しやすくするため、バグ修正タスクの前にHTMLコメントで日付と概要を書く。

### feature ファイルの TODO.md 互換設計

feature ファイルは Ralph Loop の TODO.md としても機能する。以下のルールに従う。

**チェックボックスはタスクのみ:**
- 要求・要件: チェックボックスなし（仕様記述として扱う）
- タスクの `- [ ]` / `- [x]`: そのタスクが完了したか（Ralph Loop の処理対象）

**Ralph Loop との連携:**
- `ralph` を実行するだけで、`docs/features/` の feature ファイルを自動検出し、F-xxx 番号順に順次処理する（TODO.md 不要）
- タスク行（`F-xxx-T` を含む `- [ ]` 行）のみを完了判定に使う。要求・要件のチェックボックスは無視される
- 1つの feature の全タスクが `[x]` になると、次の feature へ自動移行する
- `status: done` の feature はスキップされる
- 1ファイルだけ指定する場合: `ralph --todo docs/features/F-001_xxx.md`

**frontmatter の status:**
- `draft`: 要求・要件の定義中
- `spec`: 技術仕様の策定中
- `ready`: 実装準備完了（タスクが定義済み）
- `in-progress`: 実装中（タスクを消化中）
- `done`: 全タスク完了

**完了判定:**
- 全タスクが `[x]` になったら status を `done` に変更する
- frontmatter の progress にはタスクの完了数 / 総数を記載する（例: `3/9`）

**バグ修正時の運用:**
- バグ修正タスクは既存の feature ファイルに追記する（詳細は「バグ修正タスクの追記ルール」セクション参照）
- status は `done` → `in-progress` → `done` と遷移する（許容する）

## ドキュメント作成タイミング

| タイミング | 作成・更新するドキュメント |
|-----------|------------------------|
| 新機能の企画・ブレスト後 | docs/features/{番号}_{機能名}.md（要求・要件セクション） |
| API・DB・UI設計時 | docs/features/{番号}_{機能名}.md（技術仕様セクション）+ architecture.md |
| 技術選定・設計判断時 | docs/notes/{日付}_adr-{番号}_{タイトル}.md |
| 実装計画策定時 | docs/features/{番号}_{機能名}.md（タスクセクション） |
| タスク完了時 | docs/features/{番号}_{機能名}.md（チェックボックス更新 + frontmatter の progress/status 更新） |
| セキュリティレビュー時 | docs/notes/{日付}_security-{番号}_{テーマ}.md |
| 毎日の作業終了時 | docs/notes/{日付}_journal.md |

## ドキュメント更新トリガー

| イベント | 更新するドキュメント |
|---------|-------------------|
| 要件変更 | feature ファイル（要求・要件セクション）→ タスクセクション |
| API エンドポイント追加・変更 | feature ファイル（技術仕様 > API）+ architecture.md |
| テーブル追加・カラム変更 | feature ファイル（技術仕様 > DB）+ architecture.md |
| ライブラリ・フレームワーク変更 | notes/{日付}_adr_{タイトル}.md + architecture.md |
| アーキテクチャ変更 | notes/{日付}_adr_{タイトル}.md → 影響を受ける全ドキュメント |
| バグ修正で仕様が明確化された | feature ファイル（要件の明確化として追記） |
| 実装中に設計判断をした | notes/{日付}_journal.md → 必要なら ADR |

## コミット前チェックリスト

コードをコミットする前に、以下を確認する：

- [ ] 新機能に対応する feature ファイルが存在する
- [ ] API変更があれば feature ファイルの技術仕様と architecture.md が更新されている
- [ ] DB変更があれば feature ファイルの技術仕様と architecture.md が更新されている
- [ ] 技術的な判断をした場合は ADR（notes/）が作成されている
- [ ] タスクのチェックボックスが最新の状態になっている

## Red Flags — 以下に該当したら立ち止まること

| Red Flag | 正しい対応 |
|----------|-----------|
| 「ドキュメントは後で書く」 | ドキュメントが先。コードは後。 |
| 「コードがドキュメント」 | コードは HOW を示す。WHY と WHAT はドキュメントに書く。 |
| 「小さい変更だからドキュメント不要」 | 小さい変更の積み重ねが乖離を生む。更新する。 |
| 「feature ファイルは大げさ」 | 形式は最小限でよい。内容（何を・なぜ）が重要。 |
| 「ADRは大げさ」 | 判断の記録は将来の自分を助ける。3行でもよいから書く。 |
| 「仕様が決まっていないから書けない」 | 決まっていないことを書く。未決事項リストも立派なドキュメント。 |
| 「実装が先で、後からドキュメントを合わせる」 | 実装に引きずられたドキュメントは仕様書ではなく記録になる。 |
| 「architecture.md への反映を忘れた」 | feature に API/DB を追加したら architecture.md の一覧も更新する。 |
