# ドキュメント駆動開発（Document-Driven Development）

## 概要

コードを書く前にドキュメントを書く。コードを変えたらドキュメントも更新する。

**基本原則：** ドキュメントが存在しない機能は、合意されていない機能である。

## docs/ ディレクトリ構造

```
docs/
├── prd/                    # 製品要求仕様書（Product Requirements Document）
│   └── {feature-name}.md   # 機能ごとに1ファイル
├── api/                    # API仕様書
│   └── {endpoint-group}.md # エンドポイントグループごと
├── db/                     # データベース設計
│   └── {domain}.md         # ドメインごと
├── test-plans/             # テスト計画
│   └── {feature-name}.md   # 機能ごと
├── adr/                    # Architecture Decision Records
│   └── {number}-{title}.md # 連番-タイトル
├── ui/                     # UI設計（ui-design スキルが管理）
│   └── ...
└── journal/                # 開発ジャーナル（日付ごとの判断記録）
    └── {date}.md
```

## ドキュメント作成タイミング

| タイミング | 作成するドキュメント |
|-----------|-------------------|
| 新機能の企画・ブレスト後 | PRD（docs/prd/） |
| API設計時 | API仕様書（docs/api/） |
| データモデル設計時 | DB設計書（docs/db/） |
| テスト戦略決定時 | テスト計画（docs/test-plans/） |
| 技術選定・設計判断時 | ADR（docs/adr/） |
| UI設計時 | UI設計書（docs/ui/）— ui-design スキルに委譲 |
| 毎日の作業終了時 | ジャーナル（docs/journal/） |

## ドキュメント更新トリガー

以下のイベントが発生したら、対応するドキュメントを更新する：

| イベント | 更新するドキュメント |
|---------|-------------------|
| 要件変更 | PRD → テスト計画 → API仕様（影響があれば） |
| API エンドポイント追加・変更 | API仕様書 |
| テーブル追加・カラム変更 | DB設計書 |
| テスト方針変更 | テスト計画 |
| ライブラリ・フレームワーク変更 | ADR |
| アーキテクチャ変更 | ADR → 影響を受ける全ドキュメント |
| バグ修正で仕様が明確化された | PRD（仕様の明確化として追記） |
| 実装中に設計判断をした | ジャーナル → 必要なら ADR |

## PRD テンプレート

```markdown
# {機能名} — 製品要求仕様書

## 目的
この機能が解決する問題と、ユーザーにとっての価値。

## ユーザーストーリー
- ユーザーとして、{操作}したい。なぜなら{理由}だから。

## 機能要件
1. {具体的な要件}
2. {具体的な要件}

## 非機能要件
- パフォーマンス: {基準}
- セキュリティ: {考慮事項}
- アクセシビリティ: {基準}

## 画面・UIの概要
→ 詳細は docs/ui/ を参照（ui-design スキルで作成）

## API の概要
→ 詳細は docs/api/ を参照

## データモデルの概要
→ 詳細は docs/db/ を参照

## 受け入れ基準
- [ ] {基準1}
- [ ] {基準2}

## 未決事項
- {まだ決まっていないこと}
```

## ADR テンプレート

```markdown
# ADR-{番号}: {タイトル}

## ステータス
提案 / 承認 / 廃止 / 代替（ADR-XXX に置き換え）

## コンテキスト
この判断が必要になった背景・状況。

## 検討した選択肢

### 選択肢 A: {名前}
- メリット: 
- デメリット: 
- 将来の制約: 

### 選択肢 B: {名前}
- メリット: 
- デメリット: 
- 将来の制約: 

## 決定
選択肢 {X} を採用する。

## 理由
この選択肢を選んだ具体的な理由。

## 影響
この決定によって影響を受ける部分・今後の作業。
```

## コミット前チェックリスト

コードをコミットする前に、以下を確認する：

- [ ] 新機能に対応する PRD が存在する
- [ ] API変更があれば API仕様書が更新されている
- [ ] DB変更があれば DB設計書が更新されている
- [ ] 技術的な判断をした場合は ADR が作成されている
- [ ] テスト計画が最新の要件を反映している

## Red Flags — 以下に該当したら立ち止まること

| Red Flag | 正しい対応 |
|----------|-----------|
| 「ドキュメントは後で書く」 | ドキュメントが先。コードは後。 |
| 「コードがドキュメント」 | コードは HOW を示す。WHY と WHAT はドキュメントに書く。 |
| 「小さい変更だからドキュメント不要」 | 小さい変更の積み重ねが乖離を生む。更新する。 |
| 「PRDは形式的すぎる」 | 形式は最小限でよい。内容（何を・なぜ）が重要。 |
| 「ADRは大げさ」 | 判断の記録は将来の自分を助ける。3行でもよいから書く。 |
| 「仕様が決まっていないから書けない」 | 決まっていないことを書く。未決事項リストも立派なドキュメント。 |
| 「実装が先で、後からドキュメントを合わせる」 | 実装に引きずられたドキュメントは仕様書ではなく記録になる。 |
