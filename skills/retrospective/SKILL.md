# 振り返り（Retrospective）

## このスキルの役割

完了した作業から学びを抽出し、ドキュメントに残す。

**基本原則：振り返らないと、同じ過ちを繰り返す。**

harness は「ドキュメント駆動」を哲学とするが、ドキュメントには
**作るもの**（要求・要件・技術仕様）だけでなく、**作り方の学び**も
残す価値がある。後者を蓄積する仕組みが振り返り。

## いつ使うか

### 必須のタイミング

- **feature ファイルの status が done になったとき**
- **大きなバグを修正した直後**（フロー B 完了時）
- **仕様変更を完了したとき**（フロー E 完了時）

### 推奨のタイミング

- 開発スプリント / マイルストーンの区切り
- セキュリティレビュー指摘を多数受けた直後
- 「あれ、これ前にもやった気がする」と感じたとき
- 時間が大幅に超過した／逆に大幅に短縮できた作業の直後

### 使わないでよい場面

- 数行の typo 修正
- 自明なリファクタ
- まだ作業途中（途中の思考整理は systematic-debugging や SCRATCH.md で）

## 出力先

```
docs/notes/{日時}_retrospective_{対象}.md
```

例:
- `docs/notes/2026-05-01-1430_retrospective_F-001-user-registration.md`
- `docs/notes/2026-05-01-1500_retrospective_sprint-2026-04.md`

ファイル名命名は他の notes（journal, brainstorm, adr）と整合させる。

## KPT フォーマット

シンプルだが効く。3 セクションで構成する。

```markdown
---
date: 2026-05-01
type: retrospective
target: F-001 ユーザー登録機能
duration: 2026-04-15 〜 2026-04-30
participants: claude, ユーザー
related_features:
  - F-001
related_notes:
  - docs/notes/2026-04-17_brainstorm_F-001-設計.md
---

# 振り返り: F-001 ユーザー登録機能

## サマリ

1-3 行で「何を完成させたか」「総所感」を書く。

## Keep（続けたい・うまくいったこと）

- うまくいった点を 3-5 個、**具体的なエピソード**として書く
- 抽象論「テストを書いて良かった」ではなく
  「T03 の統合テストを書いた段階で SQL の型不整合が発覚し、
   実装前に直せたので手戻りが小さかった」のように

## Problem（うまくいかなかったこと）

- 課題を 3-5 個、**何が起きたか + 何が原因か** をセットで
- 自責も他責も避ける。事実を中立に書く
- 例: 「OAuth プロバイダの選定に時間をかけすぎた（2 日）。
       選定基準を最初に決めなかったため候補を3周比較してしまった」

## Try（次に試したいこと）

- Problem に対する具体的なアクション
- 「次の feature では選定前に基準を3個決めて、ADR に記録する」のように
  **次の作業で実行可能な形**にする
- できれば対応する Problem に番号で紐付ける

## 数値メトリクス（任意）

定量化できる学びは数値で残す：

| 指標 | 計画 | 実績 | 差分 |
|---|---|---|---|
| 工数（人日） | 5 | 7 | +2 |
| タスク数 | 12 | 15（+3 修正タスク） | +3 |
| バグ件数（リリース後） | 0 | 1 | +1 |
| テストカバレッジ | 80% | 85% | +5% |

## 関連ドキュメント

- feature ファイル: docs/features/F-001_user-registration.md
- 関連 ADR: docs/notes/2026-04-18_adr-005_oauth-provider.md
- セキュリティレビュー: docs/notes/2026-04-25_security_F-001.md
```

## 振り返りの進め方（メイン Claude 向け）

### Step 1: 振り返り対象の特定

ユーザーが「振り返り」と発したとき、まず対象を確定する：

```
Q1: 何の振り返りか？
  ├── 単一 feature の完了 → feature ID を取る
  ├── 複数 feature を含むスプリント → 期間を取る
  └── 特定のバグ・障害 → 該当 feature と影響範囲を取る
```

不明なら、最後に status: done になった feature ファイルを候補として提示する。

### Step 2: 入力情報の収集

メイン Claude が Read すべきもの：

- 対象 feature ファイル（要求・要件・技術仕様・タスクの全セクション）
- 対象期間中の `docs/notes/{日付}_*.md`（journal, brainstorm, adr, security）
- git log（対象期間のコミット履歴）
- セキュリティレビューがあればその結果

これらを横断して、Keep / Problem / Try の素材を集める。

### Step 3: ユーザーとの対話

メイン Claude が一方的に書かない。以下を必ず聞く：

1. 「{対象} の振り返りで、特に良かったと感じる点は？」
2. 「逆に、もう一度やるなら違う進め方をしたい点は？」
3. 「次の feature に持ち越す課題はある？」

ユーザーの答えと、Step 2 で集めた事実を統合して KPT を書く。

### Step 4: 出力と関連ファイルの更新

- `docs/notes/{日時}_retrospective_{対象}.md` を作成
- Try セクションの項目で「次の feature の作り方を変える」ものがあれば、
  対応するスキル（writing-plans / coding-standards / web-testing 等）への
  追記提案をユーザーに提示
- 該当 feature ファイルの末尾に「## 振り返り」セクションを追加し、
  retrospective ノートへのリンクを置く

## Red Flags

| Red Flag | 正しい対応 |
|---|---|
| 「次の feature が忙しいから振り返りは省略」 | 5 分でいいから書く。蓄積されないと再現する |
| Keep が「特になし」になった | 必ず1つは挙げる。完璧じゃないなら逆に Keep が大量にあるはず |
| Problem が他責だらけ | 自分側で次にできることを 1 つは書く |
| Try が抽象的（「もっと注意する」） | 具体的な行動に落とす（「次のレビューで X を必ず確認する」） |
| 振り返りの書き出しに 30 分以上かかった | 完璧を目指していない。3-5 個ずつで十分 |
| 過去の振り返りを読み返さない | 月1回、過去の Try が実行されたか確認する |

## 振り返りを蓄積するメリット

1. **学習速度の向上**: 同じ過ちを繰り返さない
2. **チーム知識化**: 個人の経験が docs/notes/ で共有資産になる
3. **CLAUDE.md / スキルの改善材料**: Try セクションが harness 自体への
   フィードバックになる
4. **学習者にとっての教材**: グローバル CLAUDE.md の「学習方針」と
   整合する。なぜそうなったかを後から辿れる

## 関連スキル

- `document-lifecycle`: 振り返りノートも notes/ の一種なので命名規則に従う
- `verification-before-completion`: 完了直後にこのスキルを呼ぶ流れが理想
- `claude-md-generator`: Try で「全プロジェクトに適用したいルール」が
  出たら CLAUDE.md に反映する
