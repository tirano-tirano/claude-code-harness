# Security Review Skill

コード変更に対してセキュリティチェックを実施し、問題を記録・修正するスキル。

**開始時に宣言:** 「security-review スキルでセキュリティチェックを実施します。」

**重要: フレームワーク固有のチェック項目は docs/ を参照すること。**

| 技術スタック | 参照先 |
|---|---|
| Next.js（React） | `docs/nextjs-security.md` |

---

## 全体フロー

```
1. スコープ判定（何をチェックするか）
2. 該当カテゴリのチェック実施（+ フレームワーク固有の docs/ を参照）
3. 発見事項をドキュメント化
4. 修正タスク作成 → development-cycle で修正
5. 修正後の再チェック
```

---

## スコープ判定

変更内容に基づき、チェックすべきカテゴリを選択する。

| 変更内容 | 必須チェックカテゴリ |
|---|---|
| フォーム追加・変更 | CAT-1, CAT-2, CAT-5 |
| API エンドポイント追加 | CAT-1, CAT-2, CAT-3, CAT-5, CAT-7 |
| 認証・認可ロジック変更 | CAT-1, CAT-3, CAT-7 |
| DB操作追加・変更 | CAT-2, CAT-4, CAT-5 |
| 外部API連携追加 | CAT-4, CAT-6, CAT-8 |
| 依存パッケージ追加・更新 | CAT-6 |
| デプロイ設定変更 | CAT-8, CAT-9 |
| ファイルアップロード機能 | CAT-2, CAT-5, CAT-8 |
| エラーハンドリング変更 | CAT-7, CAT-10 |
| 新規プロジェクト立ち上げ | 全カテゴリ |

## チェックカテゴリ一覧（OWASP Top 10:2025 準拠）

### CAT-1: アクセス制御の不備（A01:2025 Broken Access Control）

アクセス制御とは「誰が何をできるか」を制限する仕組み。

**チェック項目:**
- [ ] API エンドポイントの先頭で認証チェック（セッション確認）をしているか
- [ ] リソースアクセス時に「このユーザーがこのリソースを操作してよいか」を確認しているか（認可チェック）
- [ ] URL パラメータの ID を直接使ってDB検索していないか（IDOR: 他人のIDを推測してアクセス）
- [ ] 管理者機能にロールチェックがあるか
- [ ] API レスポンスに不要なフィールド（他ユーザーの情報等）を含めていないか
- [ ] ミドルウェアやフィルターだけに認証を頼っていないか（バイパスの可能性）

### CAT-2: セキュリティ設定ミス（A02:2025 Security Misconfiguration）

デフォルト設定のまま本番に出してしまう、不要な機能が有効、エラーメッセージが詳細すぎるなど。

**チェック項目:**
- [ ] 開発用の設定が本番に残っていないか（デバッグモード、verbose エラー表示）
- [ ] 不要な API エンドポイントやページが公開されていないか
- [ ] ディレクトリリスティングが無効か
- [ ] シークレットを含むファイルが .gitignore に含まれているか
- [ ] クライアントに公開される環境変数にシークレットを入れていないか
- [ ] エラーページがスタックトレースやDB情報を表示しないか
- [ ] Source Map が本番で無効か

### CAT-3: ソフトウェアサプライチェーンの問題（A03:2025 Software Supply Chain Failures）

使っているライブラリや依存パッケージ経由で脆弱性が入り込むリスク。

**チェック項目:**
- [ ] 依存パッケージの脆弱性スキャンを実行したか
- [ ] ロックファイルがコミットされているか（依存バージョンの固定）
- [ ] 新規パッケージ追加時にダウンロード数・メンテナンス状況を確認したか
- [ ] typosquatting（似た名前の悪意あるパッケージ）に注意しているか
- [ ] 依存パッケージの自動更新が設定されているか
- [ ] postinstall スクリプトが意図しない動作をしないか確認したか

### CAT-4: 暗号化の問題（A04:2025 Cryptographic Failures）

パスワードやトークンなど機密データの保護が不十分。

**チェック項目:**
- [ ] パスワードを平文保存していないか（bcrypt / argon2 等でハッシュ化）
- [ ] HTTPS が強制されているか（HSTS ヘッダー設定）
- [ ] セッショントークンが十分なランダム性を持っているか
- [ ] API キーや認証トークンがコードにハードコードされていないか
- [ ] Cookie に `Secure`, `HttpOnly`, `SameSite` 属性が設定されているか
- [ ] JWT を使っている場合、適切なアルゴリズムと有効期限が設定されているか
- [ ] 機密データがクライアント側に漏洩していないか

### CAT-5: インジェクション（A05:2025 Injection）

外部からの入力値がコードやクエリとして実行されてしまう攻撃。

**チェック項目:**
- [ ] ユーザー入力をスキーマで検証しているか
- [ ] SQL を直接組み立てていないか（ORM / プリペアドステートメントを使う）
- [ ] HTML にユーザー入力を埋め込む際にサニタイズしているか
- [ ] URL パラメータをそのまま DB クエリやファイルパスに使っていないか
- [ ] `eval()` 相当の動的コード実行を使っていないか
- [ ] テンプレートに未検証の外部入力を埋め込んでいないか

### CAT-6: 安全でない設計（A06:2025 Insecure Design）

実装のバグではなく、設計段階でセキュリティが考慮されていない問題。

**チェック項目:**
- [ ] レート制限（一定時間内のリクエスト回数制限）が実装されているか
- [ ] ブルートフォース攻撃（総当たり攻撃）への対策があるか
- [ ] 機密操作に二重確認（パスワード再入力等）があるか
- [ ] ビジネスロジックに「本来ありえない操作」を防ぐガードがあるか
- [ ] エラーメッセージが情報漏洩しないか
- [ ] 並行処理での競合状態（Race Condition）を考慮しているか

### CAT-7: 認証の問題（A07:2025 Authentication Failures）

ログインや本人確認の仕組みが不十分。

**チェック項目:**
- [ ] パスワード強度のバリデーションがあるか
- [ ] セッション管理が適切か（タイムアウト設定、ログアウト時の無効化）
- [ ] CSRF 対策が実装されているか
- [ ] OAuth / ソーシャルログインの state パラメータを検証しているか
- [ ] パスワードリセットフローが安全か（トークンの有効期限、使い捨て）
- [ ] セッションIDが認証成功後に再生成されるか
- [ ] 多要素認証（MFA）の導入を検討したか

### CAT-8: ソフトウェア・データ完全性の問題（A08:2025 Software or Data Integrity Failures）

更新や CI/CD パイプラインの信頼性が担保されていない。

**チェック項目:**
- [ ] CDN から読み込む外部スクリプトに SRI（Subresource Integrity）ハッシュを付けているか
- [ ] CI/CD パイプラインでの権限が最小限か
- [ ] デプロイ時にビルド成果物の整合性を検証しているか
- [ ] サードパーティの CI アクションをピン留め（SHA指定）しているか

### CAT-9: ログと監視の問題（A09:2025 Security Logging & Alerting Failures）

攻撃の検知や事後調査に必要なログが取れていない。

**チェック項目:**
- [ ] 認証の成功・失敗をログに記録しているか
- [ ] 権限エラー（403）をログに記録しているか
- [ ] ログに機密情報（パスワード、トークン、個人情報）を含めていないか
- [ ] ログの改ざん防止策があるか
- [ ] 異常検知のアラートが設定されているか

### CAT-10: 例外処理の不備（A10:2025 Mishandling of Exceptional Conditions）

エラー時にシステムが「安全でない状態」に陥る問題。2025年の新カテゴリ。

**チェック項目:**
- [ ] try-catch で適切にエラーを捕捉しているか
- [ ] エラー時に「安全側に倒す」設計になっているか（fail-safe）
- [ ] エラーページが適切に設定されているか
- [ ] エラーレスポンスがクライアントに詳細を漏らさないか
- [ ] 外部サービス障害時のフォールバック処理があるか
- [ ] 予期しない入力型に対するガードがあるか

## インフラ・デプロイ設定チェック

### HTTP セキュリティヘッダー

以下のヘッダーが設定されているか確認する（設定方法はフレームワーク・サーバーによって異なる）:

| ヘッダー | 推奨値 | 目的 |
|---|---|---|
| Strict-Transport-Security | `max-age=63072000; includeSubDomains; preload` | HTTPS 強制 |
| X-Content-Type-Options | `nosniff` | MIME タイプスニッフィング防止 |
| X-Frame-Options | `DENY` | クリックジャッキング防止 |
| Referrer-Policy | `strict-origin-when-cross-origin` | リファラー情報の制限 |
| Permissions-Policy | `camera=(), microphone=(), geolocation=()` | ブラウザ機能の制限 |
| Content-Security-Policy | プロジェクトに応じて設定 | XSS・コードインジェクション防止 |

**チェック項目:**
- [ ] 上記ヘッダーがすべて設定されているか
- [ ] CSP で unsafe-inline / unsafe-eval を避けているか
- [ ] CORS が全許可（`*`）になっていないか
- [ ] Cookie の属性（SameSite, Secure, HttpOnly）が適切か

### デプロイ設定

- [ ] 環境変数が本番・開発で分離されているか
- [ ] 本番の環境変数に開発用の値が残っていないか
- [ ] WAF（Web Application Firewall）の導入を検討したか
- [ ] DDoS 対策（レート制限、CDN）が設定されているか

**フレームワーク固有のインフラ設定は docs/ を参照。**

## 発見事項のドキュメント化

チェックで問題を発見した場合、以下の形式で `docs/security-findings.md` に記録する。

```markdown
# セキュリティ発見事項

## [日付] セキュリティレビュー

### 概要
- レビュー対象: [変更内容の説明]
- チェック範囲: [CAT-1, CAT-5, CAT-8 等]
- 発見件数: 重大 X件 / 警告 Y件 / 情報 Z件

### 発見事項

#### [重大] SEC-001: APIエンドポイントに認証チェックなし
- カテゴリ: CAT-1（アクセス制御）
- ファイル: [ファイルパス]
- 内容: [具体的な問題]
- 影響: [起きうる被害]
- 修正方針: [どう直すか]
- ステータス: ⬜ 未修正 / ✅ 修正済み
```

### 重大度の判定基準

| 重大度 | 基準 | 例 |
|---|---|---|
| 重大 | データ漏洩・改ざん・認証バイパスの可能性 | 認証チェック漏れ、SQLインジェクション |
| 警告 | 攻撃の足がかりになりうる | CSPヘッダー未設定、エラー詳細表示 |
| 情報 | ベストプラクティスから外れている | Source Map有効、不要なAPIエンドポイント |

## 修正フロー

```
発見事項を記録
  ↓
重大度「重大」→ 即座に development-cycle で修正
重大度「警告」→ 次のスプリントまでに修正
重大度「情報」→ バックログに追加
  ↓
修正完了後、該当カテゴリを再チェック
  ↓
docs/security-findings.md のステータスを更新
```

セキュリティ修正は development-cycle の「バグ修正フロー（フロー B）」を使用する。
コミットメッセージには `fix(security):` プレフィックスを使用。

## 定期チェック（プロジェクト全体）

- [ ] 依存パッケージの脆弱性スキャン（週1回以上推奨）
- [ ] 依存パッケージの更新確認
- [ ] 環境変数の棚卸し（不要なものの削除）
- [ ] アクセスログの異常確認
- [ ] セキュリティヘッダーの検証（https://securityheaders.com/ 等）
- [ ] SSL/TLS 証明書の有効期限確認

## Red Flags

以下を検出したら **即座に指摘** すること:

1. ❌ API エンドポイントに認証チェックがない
2. ❌ ミドルウェア・フィルターのみで認証を制御している
3. ❌ HTML にサニタイズなしのユーザー入力を埋め込んでいる
4. ❌ SQL を文字列結合で組み立てている
5. ❌ クライアント公開の環境変数にシークレットが入っている
6. ❌ 動的コード実行（eval 相当）を使っている
7. ❌ パスワードを平文で保存・比較している
8. ❌ CORS が全許可になっている
9. ❌ エラーレスポンスにスタックトレースやDB情報が含まれている
10. ❌ 重大な脆弱性がある状態でデプロイ
