# ブレインストーミング → ドキュメント出力

## 概要

考えたことを空中に消さない。ブレインストーミングの成果を必ずドキュメントに残す。

**基本原則：** ブレインストーミングのゴールは「話し合うこと」ではなく「判断可能な状態を作ること」。


## 3 つの Phase

```
Phase 1: 準備 → Phase 2: 対話しながら整理する → Phase 3: ドキュメント出力
```

Phase 2 が中核。対話とメモの構造化を同時に行い、会話の終わりにはメモがほぼ整理済みの状態になっている。

---

### Phase 1: 準備

1. **出力先を決める** — ブレスト結果をどのドキュメントにするか最初に決める
2. **構造化メモファイルを作成** — `docs/notes/{日付}_brainstorm_{テーマ}.md` を以下の構造で作成する

出力先の対応表：

| ブレストの目的 | 出力先ドキュメント |
|--------------|-------------------|
| 新機能の企画 | docs/features/{番号}_{機能名}.md（要求・要件セクション） |
| 技術選定 | docs/notes/{日付}_adr-{番号}_{タイトル}.md |
| UI設計の方向性 | docs/features/{番号}_{機能名}.md（技術仕様 > UI）→ ui-design スキルに引き継ぎ |
| テスト戦略 | docs/features/{番号}_{機能名}.md（要件からテストケースを導出） |
| バグの原因調査 | docs/notes/{日付}_journal.md |
| その他 | docs/notes/{日付}_brainstorm_{テーマ}.md |

**構造化メモの初期構造：**

```markdown
# ブレインストーミング — {テーマ}

日時: {date}
出力先: {Phase 1 で決めた出力先}

## 決定事項
（対話で合意が取れたものをここに移動する）

## 検討中
（現在議論しているトピック）

## 未決事項
（結論が出なかった・情報不足で保留にしたもの）

## 捨てた案
（検討したが採用しなかった案。理由も記録する）
```

---

### Phase 2: 対話しながら整理する

対話とメモの構造化を**同時に**行う。質問→回答→メモ更新を1サイクルとして繰り返す。

**1サイクルの流れ：**

```
質問する（1回に1つ）
  ↓
ユーザーの回答を受け取る
  ↓
メモファイルを更新する:
  - 合意が取れた → 「決定事項」に追加
  - まだ議論中 → 「検討中」に追加または更新
  - 保留にした → 「未決事項」に移動
  - 却下された → 「捨てた案」に移動（理由付き）
  ↓
次の質問へ（または区切りの確認）
```

**対話のガイドライン：**

| やること | やらないこと |
|---------|------------|
| 1回に1つの質問をする | 5個の質問を一度に投げない |
| 相手の回答を受けて深掘りする | 自分の結論に誘導しない |
| 「なぜ？」「誰のため？」を聞く | 「こうすべき」と断定しない |
| 選択肢を提示して選んでもらう | 1つの案だけ提示しない |
| 回答のたびにメモを更新する | 対話が終わってからまとめない |

**メモ更新のルール：**

- 「検討中」セクションは常に最新の議論状態を反映する。結論が出たら「決定事項」か「捨てた案」に移動し、「検討中」からは消す
- 「決定事項」に書く内容は、結論だけでなく「なぜそう決めたか」の理由も1行で添える
- 「捨てた案」に書く内容は、案の概要と「なぜ捨てたか」の理由。後で同じ案が再浮上したときに判断を繰り返さないため
- 「未決事項」には、何が分かれば判断できるか（必要な情報・条件）も書く

**区切りの確認（3〜5サイクルごと）：**

対話が3〜5サイクル進んだら、メモの現在の状態をユーザーに見せて確認する。

```
ここまでの整理状況です：
- 決定事項: N 件
- 検討中: N 件
- 未決事項: N 件
この内容で合っていますか？ 修正や追加はありますか？
```

ユーザーの修正を反映してから次に進む。

---

### Phase 3: ドキュメント出力

Phase 2 でメモが整理済みなので、出力先に転記する作業が中心。

**feature ファイルを出力する場合：**

構造化メモの各セクションを以下のように変換する：

| メモのセクション | feature ファイルの対応先 |
|----------------|----------------------|
| 決定事項（目的・ストーリーに関するもの） | 要求（R01, R02...） |
| 決定事項（具体的条件に関するもの） | 要件（S01, S02...） |
| 決定事項（技術方針に関するもの） | 技術仕様（API/DB/UI） |
| 未決事項 | 未決事項セクション |
| 捨てた案 | メモファイルに残す（feature ファイルには含めない） |

**feature ファイルのテンプレート：**

```markdown
---
id: F-001
feature: {機能名}
status: draft
progress: 0/0
---

## 要求

- F-001-R01: {ユーザー視点の目的・ストーリー1}
- F-001-R02: {回答2}

## 要件

- F-001-S01: {要求を分解した具体的条件。決定事項から抽出。} （参照: R01）
- F-001-S02: {同上} （参照: R01）

（ブレスト段階で分解しきれない場合は、要求だけ書いて要件は後から追加してよい。）

## 技術仕様

### API
（ブレストで方向性が決まっていれば管理番号付きで記載。未定なら「未定」と明記。）

### DB
（同上）

### UI
（同上）

## タスク
（まだ実装計画が立っていなければ空欄でよい。writing-plans スキルで後から策定。）
（策定時は外側→内側のテスト駆動: 受け入れテスト→統合テスト→単体テスト→実装の順。）

## 未決事項
（未決事項をそのまま記載 — 隠さない）
```

**ブレスト後の次のステップ：**
- feature ファイルの未決事項を解消
- 技術仕様セクションを詳細化（document-lifecycle スキル）
- UI設計を詳細化（ui-design スキル）
- タスクセクションを策定（writing-plans スキル）— 外側→内側のテスト駆動: 受け入れテスト→統合テスト→単体テスト→実装
- architecture.md に横断情報を反映
- Ralph Loop で自動実行: `ralph --todo docs/features/F-xxx_yyy.md`（feature ファイルは自動検出される）

---

## Red Flags — 以下に該当したら立ち止まること

| Red Flag | 正しい対応 |
|----------|-----------|
| 「対話だけ進めてメモを更新していない」 | 回答のたびにメモを更新する。3サイクル以上メモ未更新は禁止。 |
| 「検討中セクションが膨らみ続けている」 | 結論が出たものを「決定事項」か「捨てた案」に移動する。「検討中」は今議論しているものだけ。 |
| 「ブレスト結果をドキュメントにしなかった」 | Phase 3 を実行する。ドキュメントにならないブレストは無かったのと同じ。 |
| 「結論が出なかったからドキュメントに書けない」 | 未決事項としてドキュメントに書く。「決まっていない」ことも記録すべき情報。 |
| 「1人で考えて答えを出した」 | ソクラテス式対話に戻る。質問して相手に考えてもらう。 |
| 「提案を5個並べて選ばせた」 | 1つずつ質問で深掘りする。選択肢の羅列は対話ではない。 |
| 「feature ファイルのフォーマットに合わないから書かない」 | フォーマットは調整してよい。内容が残ることが重要。 |
| 「捨てた案の理由を書かなかった」 | 理由なしで案を消すと、後で同じ議論を繰り返す。理由を1行でも書く。 |
