# 開発サイクル（Development Cycle）

## このスキルの役割

コード変更を伴う作業を始めるとき、**最初にこのスキルを読んで、今の作業がどのフローに該当するかを判定する。** 各フローには「どのスキルを、どの順番で使うか」と「どこでドキュメントを更新するか」が定義されている。

**基本原則：フローに沿って進む。フローを飛ばさない。**

## フロー判定

作業を始める前に、以下のどれに該当するかを判定する：

| ユーザーの指示の特徴 | 該当フロー |
|-------------------|-----------|
| 新しい機能を作る、追加する | **A: 新機能開発** |
| バグを直す、エラーを修正する | **B: バグ修正** |
| リファクタリング、整理、改善（機能変更なし） | **C: リファクタリング** |
| 画面を作る、UIを変える、デザインを変更する | **D: UI改修** |
| 仕様が変わった、要件を変更する | **E: 仕様変更** |

複数に該当する場合（例：「新機能を追加して、UIも作って」）は、**もっとも広いフロー**（この場合は A: 新機能開発）を選ぶ。A が最も広く、B が最も狭い。

判定できない場合は、ユーザーに確認する。推測で進めない。

## 全フロー共通ルール

どのフローを通っても、以下は**必ず守る**。これらはフロー内の個別 Step より優先される。

### 1. ドキュメントを先に書く（ドキュメント駆動）

```
コードを書く前に、何を・なぜ作るのかをドキュメントに書くこと
```

**必要なドキュメント：**
- 新機能 → docs/features/{番号}_{機能名}.md が必須（要求・要件・技術仕様・タスク + 関連 NFR）
- 共通基盤 → docs/features/F-000-{番号}_{基盤名}.md（対応する NFR を参照）
- API・DB変更 → feature ファイルの技術仕様セクション + docs/architecture.md を更新
- UI変更 → feature ファイルの技術仕様 > UI セクション + docs/design-system.md との整合性確認
- 技術判断 → docs/notes/{日時}_adr-{番号}_{タイトル}.md を作成
- 非機能要求の追加・変更 → docs/nfr/{カテゴリ}.md を更新

**タイミング：**
- 新規ドキュメントは**実装開始前**に作成する
- 既存ドキュメントの更新は**コミット前**に行う
- 「ドキュメントは後で書く」は禁止。後では書かない

**ドキュメントが存在しない機能は、合意されていない機能である。**

詳細は **document-lifecycle** スキルに従う。

### 2. テストを先に書く（TDD 鉄則）

```
本番コードを書く前に、必ず失敗するテストを先に書くこと
```

すべての Phase、すべてのフローに適用される。例外はない。

**Red-Green-Refactor サイクル：**
1. **RED** — テストを1つ書く → 実行 → **失敗を確認**
2. **GREEN** — テストを通す最小限のコードだけ書く → 実行 → **成功を確認**
3. **REFACTOR** — テスト緑のまま、コードを整理

**各ステップで必ずテストを実行して結果を確認すること。**

**実装コードを先に書いてしまった場合：**
- **その実装コードを削除して、テストから書き直す**
- 「参考として残す」は禁止 — 見ながら書くとテスト後付けと同じ結果になる
- 「もったいない」は錯覚 — 検証されていないコードを残す方がリスクが高い
- 「手動で確認済み」は不十分 — 手動テストは記録に残らず再実行できない

**モックの使用制限：**
- 外部 API（Stripe, SendGrid 等）→ ✅ 許可
- データベース → ❌ 禁止（実DBでテスト。Docker等でテスト用DB起動）
- 内部 API Route → ❌ 禁止（実際のルートハンドラを通す）
- ブラウザ操作 → ❌ 禁止（実ブラウザでE2E）
- 詳細は **web-testing** スキルのモック許可制度セクションに従う

### 3. コミット・ブランチは規約に従う

```
コミットメッセージは Conventional Commits 形式で書く
ブランチ名は命名規則に従う
```

詳細は **git-conventions** スキルに従う。

### 4. 完了前に検証する

作業完了を宣言する前に、以下を確認する：
- すべてのテストが通る
- 新規・変更コードにテストが存在する
- 影響を受けるドキュメントが更新されている（feature ファイル + architecture.md）
- テスト出力にエラーや警告がない
- feature ファイルのタスクチェックボックスと frontmatter（progress, status）を更新
- feature ファイルの関連 NFR セクションに列挙された NFR の遵守チェックリストを確認（docs/nfr/ 参照）
- UI 変更がある場合は docs/design-system.md の共通パターンとの整合性を確認
- セキュリティに影響する変更がある場合は **security-review** スキルでチェック
- CLAUDE.md の更新が必要か確認（新機能追加、技術スタック変更、外部サービス追加等があった場合は **claude-md-generator** スキルで更新）
- PR作成が必要な場合は **remote-repository** スキルのテンプレートに従う

詳細は **verification-before-completion** スキルに従う。

### 5. 開発途中で計画が変わったときのルール

開発の途中で、新たな機能の必要性が判明したり、既存の仕様を変更・具体化する必要が出ることがある。そのとき「何を、どの順番でやるか」を場当たり的に判断せず、以下のルールに従う。

#### 計画変更が発生するケース

| ケース | 具体例 | 対応方針 |
|--------|--------|----------|
| **A. 新機能の追加** | F-001 実装中に F-002 が必要と判明 | 新しい feature ファイルを作る |
| **B. 仕様の変更** | 要件 S03 を変えたい、API 設計を変更する | 既存の feature ファイルを更新する |
| **C. 抽象的な機能の具体化** | 「検索できる」→「全文検索 + フィルタリング」に明確化 | 要求・要件・技術仕様を追記・分割する |

#### Step 1: 作業を止めて状況を記録する

計画変更に気づいた時点で、**今やっているタスクを区切りのいいところまで完了させてから止める**。

やること：
- 現在のタスクが途中なら、コミットできる状態まで進めてコミットする
- feature ファイルの progress を現時点の実態に合わせて更新する
- 変更の内容を SCRATCH.md（Ralph Loop 使用時）またはユーザーへの報告に記録する

やらないこと：
- タスクの途中で放置して別の作業に移る
- 「後で戻る」つもりで記録せずに進む

#### Step 2: 変更の影響範囲を判定する

変更が、今作業中の feature にどう影響するかを判定する。

```
Q1: 変更は今の feature（F-001）の中の話か、別の feature（F-002）か？
  ├── 別の feature → ケース A（新機能追加）→ Step 3A へ
  └── 今の feature の中の話
        Q2: 既存の要求・要件の内容が変わるか？
          ├── はい → ケース B（仕様変更）→ Step 3B へ
          └── いいえ（曖昧だったものが明確になっただけ）
                → ケース C（具体化）→ Step 3C へ
```

判定できない場合は、ユーザーに確認する。

#### Step 3A: 新機能の追加

今の feature とは別の feature ファイルを新規作成する。

**今の feature（F-001）をどうするか：**

```
Q: F-002 がないと F-001 の実装を進められないか？
  ├── いいえ（独立している）
  │     → F-001 を続行。F-002 は後で着手する。
  │       F-001 の feature ファイルの「## 未決事項」に
  │       「F-002 との連携は F-002 完了後に対応」と記録する。
  │
  └── はい（F-002 に依存している）
        Q: F-001 の残タスクのうち、F-002 なしで進められるものはあるか？
          ├── ある → 進められるタスクだけ先に消化し、
          │         依存タスクには「BLOCKED: F-002 待ち」と注記する。
          │         その後 F-002 の作成に移る。
          │
          └── ない → F-001 の status を blocked に変更し、
                      F-002 の作成・実装を先に行う。
```

**F-002 の作成手順：**
1. フロー A の Step 1（要件定義）から開始する
2. F-002 の feature ファイルを作成する（document-lifecycle）
3. F-001 との依存関係を、両方の feature ファイルの「## 未決事項」に記録する

#### Step 3B: 仕様の変更

既存の feature ファイルの要求・要件・技術仕様を更新する。

**ドキュメントの更新（コード修正より先に行う）：**
1. feature ファイルの該当する要求（R01 等）・要件（S01 等）・技術仕様（API-01 等）を修正する
2. 変更理由を「## 変更履歴」に記録する（日付、何を、なぜ変えたか）
3. architecture.md に影響がある場合はそちらも更新する

**完了済みタスクへの影響判定：**

```
完了済みタスクのうち、変更された要求・要件・技術仕様を参照しているものを洗い出す。

各タスクについて：
  Q: そのタスクの成果物（テスト or 実装）は、変更後の仕様でも正しいか？
    ├── 正しい → そのまま。何もしない。
    ├── テストの期待値だけ変わる → 修正タスクを追記する
    │     例: - [ ] F-001-T13: T03 のテストを新仕様に合わせて修正する （参照: S03）
    └── 実装自体のやり直しが必要 → 修正タスクを追記する
          例: - [ ] F-001-T14: T08 の実装を新仕様に合わせて修正する （参照: API-01）
```

**タスクの追記ルール：**
- 完了済みタスクのチェックを外す（`[x]` → `[ ]` に戻す）のは**禁止**。完了の記録は残す。
- 代わりに、修正タスクを**末尾の検証タスクの直前**に追記する
- 追記したタスクの番号は、既存の最大番号の続きにする（例: T12 まであれば T13 から）
- progress を更新する（タスク総数が増える）

**未着手タスクへの影響：**
- 変更された仕様を参照するタスクの記述を、新しい仕様に合わせて書き換える
- タスクの順序変更が必要な場合は、writing-plans のルール（外側→内側）に従って並べ替える

#### Step 3C: 抽象的な機能の具体化

曖昧だった要求・要件が明確になった場合。仕様変更（Step 3B）と似ているが、「変更」ではなく「追加・詳細化」が主な操作になる。

**ドキュメントの更新：**
1. 抽象的だった要求（例: R01「ユーザーが検索できる」）を具体化する
   - 元の要求を修正するか、サブ要求に分割する
   - 分割例: R01 → R01「全文検索ができる」+ R01a「フィルタリングができる」
2. 具体化に伴い、要件（S01, S02...）を追加する
3. 技術仕様（API-01, DB-01...）を追加する
4. 変更履歴に「具体化」の記録を残す

**タスクの追加：**
- 新しい要件・技術仕様に対応するタスクを、writing-plans の手順で追加する
- 既存タスクとの順序関係を確認し、外側→内側の原則を維持する
- 既存の受け入れテスト・統合テストが具体化後の仕様をカバーしていない場合は、テスト修正タスクも追記する

**粒度の判断：**
```
Q: 具体化によってタスク総数が 20 を超えるか？
  ├── いいえ → そのまま追記
  └── はい → feature ファイルの分割を検討する（document-lifecycle）
```

#### Step 4: 計画変更後の再開

計画変更の処理が終わったら、以下を確認してから作業を再開する：

- feature ファイルの frontmatter（status, progress）が実態と一致している
- 未完了タスクの順序が外側→内側の原則を維持している
- 追記したタスクに参照（`参照: R01, S01`）が付いている
- 依存関係（他の feature への BLOCKED 注記等）が記録されている

確認後、次の未完了タスクから作業を再開する。Ralph Loop は `- [ ] F-xxx-Txx` を上から順に処理するので、追記タスクの位置に注意する。

## フロー A: 新機能開発

**全体の流れ：** 要件を固める → 設計する → 計画を立てる → 実装する → 検証する

```
Step 1: 要件定義
│  スキル: brainstorming-with-docs → document-lifecycle
│  成果物: docs/features/{番号}_{機能名}.md（要求・要件セクション）
│  ドキュメント: feature ファイルを作成
│
Step 2: 設計
│  ├── API設計 → feature ファイルの技術仕様 > API + architecture.md
│  ├── DB設計 → feature ファイルの技術仕様 > DB + architecture.md
│  ├── UI設計 → ui-design スキル → feature ファイルの技術仕様 > UI
│  ├── ディレクトリ構成 → project-structure スキルに従う
│  └── 技術判断 → docs/notes/{日時}_adr-{番号}_{タイトル}.md
│  スキル: document-lifecycle, ui-design, project-structure
│  ドキュメント: feature ファイルの技術仕様セクション + architecture.md を作成・更新
│
Step 3: 実装計画の策定
│  スキル: writing-plans
│  手順: 要求・要件・技術仕様からタスクを導出し、feature ファイルのタスクセクションに書き出す
│  順序: 受け入れテスト → 統合テスト → 単体テスト → 実装 → 検証（外側→内側のテスト駆動）
│  成果物: feature ファイルのタスクセクション（F-xxx-Txx 形式のチェックボックス）
│  frontmatter: status を ready に、progress を 0/{タスク総数} に更新
│
Step 4: 実装（TDD サイクル）
│  スキル: web-testing, executing-plans（または Ralph Loop / subagent-driven-development）
│  手順: タスクを上から順に処理 → テストを書く → 失敗確認 → 最小実装 → 成功確認 → リファクタ → 繰り返し
│  進捗: feature ファイルのタスクチェックボックスを完了ごとに更新
│
Step 5: 検証・完了
│  スキル: security-review, verification-before-completion, web-testing（Phase 4-5）, claude-md-generator
│  セキュリティ: 変更内容に応じたカテゴリをチェック（security-review のスコープ判定に従う）
│  ドキュメント: feature ファイル・architecture.md が最新か確認
│  進捗: feature ファイルの frontmatter（progress, status）を更新
│  CLAUDE.md: 新規プロジェクトの場合は claude-md-generator で生成
│  Git: git-conventions に従いコミット、remote-repository に従いPR作成
```

### フロー A のチェックポイント

- [ ] feature ファイルの要求・要件セクションが存在する（Step 1）
- [ ] feature ファイルの技術仕様セクションと architecture.md が存在する（Step 2）
- [ ] feature ファイルのタスクセクションが外側→内側の順序で書かれている（Step 3）
- [ ] すべてのテストが通る（Step 4）
- [ ] ドキュメントがコードと一致し、進捗が更新されている（Step 5）

## フロー B: バグ修正

**全体の流れ：** feature ファイルにタスク追記 → 再現テストを書く → 原因を特定する → 修正する → 回帰テストを確認する

```
Step 0: feature ファイルにバグ修正タスクを追記する
│  スキル: document-lifecycle（バグ修正タスクの追記ルール）
│  手順: バグが属する feature ファイルのタスクセクションに再現テスト + 修正のタスクを追記
│  frontmatter: status を in-progress に、progress のタスク総数を更新
│  ※ バグがどの feature に属するか不明な場合は、原因調査（Step 2）を先に行い、
│    特定後に追記する
│
Step 1: バグの再現テストを書く
│  スキル: web-testing
│  手順: バグを再現するテストを書く → テスト実行 → 失敗することを確認
│  ※ テストが失敗しない場合、バグの理解が不十分。再調査する。
│
Step 2: 原因を特定する
│  スキル: systematic-debugging
│  ※ 原因特定のためにコードを変更する場合も、テストを先に書く
│
Step 3: 修正する
│  手順: 最小限の修正 → テスト実行 → 成功確認
│  ※ 修正範囲を広げすぎない。バグ修正に関係ない改善は別作業にする。
│
Step 4: 回帰テスト・完了
│  スキル: verification-before-completion
│  手順: 既存テストがすべて通ることを確認
│  ドキュメント: バグが仕様の曖昧さに起因する場合、feature ファイルの要件を明確化
│  frontmatter: 全タスク完了なら status を done に戻す
```

### フロー B のチェックポイント

- [ ] feature ファイルにバグ修正タスクが追記されている（Step 0）
- [ ] バグを再現するテストが存在する（Step 1）
- [ ] テストが修正前に失敗し、修正後に成功する（Step 3）
- [ ] 既存テストがすべて通る（Step 4）
- [ ] 仕様の曖昧さがあれば feature ファイルを更新済み（Step 4）
- [ ] frontmatter の status と progress が最新（Step 4）

---

## フロー C: リファクタリング

**全体の流れ：** 既存テストを確認する → リファクタする → テストが緑のままであることを確認する

```
Step 1: 既存テストの確認
│  手順: 対象コードのテストを実行 → すべて通ることを確認
│  ※ テストが不足している場合、先にテストを追加する（テスト追加はリファクタではない）
│
Step 2: リファクタリング
│  手順: 小さい単位で変更 → テスト実行 → 緑を確認 → 繰り返し
│  ※ 機能変更を混ぜない。「ついでにこの機能も」は禁止。別作業にする。
│
Step 3: 完了
│  スキル: verification-before-completion
│  手順: すべてのテストが通ることを確認
│  ドキュメント: アーキテクチャ変更があれば docs/notes/{日時}_adr-{番号}_{タイトル}.md を作成
```

### フロー C のチェックポイント

- [ ] リファクタ前にテストがすべて通る（Step 1）
- [ ] リファクタ後にテストがすべて通る（Step 2）
- [ ] 機能変更が混ざっていない（Step 2）
- [ ] アーキテクチャ変更があれば ADR が存在する（Step 3）

---

## フロー D: UI改修

**全体の流れ：** UI設計する → 計画を立てる → 実装する → ビジュアルテストを確認する

```
Step 1: UI設計
│  スキル: ui-design, document-lifecycle
│  成果物: feature ファイルの技術仕様 > UI セクション
│
Step 2: 実装計画の策定
│  スキル: writing-plans
│  手順: UI仕様からタスクを導出（E2Eテスト → 統合テスト → コンポーネント実装）
│  成果物: feature ファイルのタスクセクション
│
Step 3: 実装（TDD サイクル）
│  スキル: web-testing, executing-plans（または Ralph Loop）
│  手順: タスクを上から順に処理 → テストを書く → 失敗確認 → UI実装 → 成功確認
│
Step 4: ビジュアル・完了
│  スキル: web-testing（ビジュアルテスト）, verification-before-completion
│  ドキュメント: feature ファイルの UI セクションが実装と一致しているか確認
│  進捗: feature ファイルの frontmatter（progress, status）を更新
```

### フロー D のチェックポイント

- [ ] feature ファイルの UI セクションが存在する（Step 1）
- [ ] feature ファイルのタスクセクションが存在する（Step 2）
- [ ] ビジュアルテストが通る（Step 4）
- [ ] feature ファイルの UI セクションが実装と一致している（Step 4）

## フロー E: 仕様変更

**全体の流れ：** 変更内容を整理する → 影響範囲を特定する → ドキュメントを更新する → 計画を立てる → テスト更新・実装修正する

```
Step 1: 変更内容の整理
│  スキル: brainstorming-with-docs（必要に応じて）
│  手順: 何が変わるのか、なぜ変わるのかを明確にする
│
Step 2: 影響範囲の特定
│  手順: 変更が影響する画面、API、DB、テストを洗い出す
│  成果物: 影響範囲リスト
│
Step 3: ドキュメント更新
│  スキル: document-lifecycle
│  手順: feature ファイル（要求→要件→技術仕様）→ architecture.md の順に更新
│  ※ ドキュメントを先に更新する。コード修正はその後。
│
Step 4: 実装計画の策定
│  スキル: writing-plans
│  手順: 変更された仕様からタスクを導出し、feature ファイルのタスクセクションを更新
│  ※ 既存タスクがある場合は追記・修正。影響範囲に応じてテスト修正→実装修正の順
│
Step 5: テスト更新・実装修正・完了
│  スキル: web-testing, executing-plans（または Ralph Loop）
│  手順: 新しい仕様に合わせてテストを修正 → 失敗確認 → 実装を修正 → 成功確認
│  スキル: verification-before-completion
│  ドキュメント: feature ファイル・architecture.md が新仕様と一致しているか確認
│  進捗: feature ファイルのタスクチェックボックスと frontmatter（progress, status）を更新
```

### フロー E のチェックポイント

- [ ] 変更理由が明確になっている（Step 1）
- [ ] 影響範囲が特定されている（Step 2）
- [ ] 影響を受ける feature ファイルと architecture.md がすべて更新されている（Step 3）
- [ ] テストが新仕様を反映している（Step 4）
- [ ] すべてのテストが通る（Step 5）

## Red Flags — 以下に該当したら立ち止まること

| Red Flag | 正しい対応 |
|----------|-----------|
| フローを判定せずに作業を始めた | 立ち止まる。まずフローを判定する。 |
| 「小さい修正だからフロー不要」と思った | 小さい修正でもフロー B（バグ修正）または C（リファクタ）に該当する。 |
| ドキュメントを更新せずにコミットしようとした | コミット前にドキュメントを更新する。 |
| テスト計画なしに実装を始めた | Step を戻る。テスト設計が先。 |
| feature ファイルなしに新機能の実装を始めた | Step 1 に戻る。要件定義が先。 |
| リファクタリング中に機能変更を混ぜた | 機能変更を取り消す。別作業として切り出す。 |
| 仕様変更なのにコードだけ修正した | ドキュメント更新が先。Step 3 に戻る。 |
| 「このフローは大げさ」と思った | フローの各 Step は省略不可。ただし成果物の量は作業規模に合わせてよい。 |
| 複数フローを同時に進めようとした | 1つのフローを完了してから次に進む。 |
| 計画変更に気づいたのにタスクの途中で別作業に移った | まず現タスクを区切りまで完了させる。共通ルール5に従う。 |
| 完了済みタスクのチェックを外して「やり直し」にした | チェックは外さない。修正タスクを追記する。共通ルール5 Step 3B に従う。 |
| 仕様が曖昧なまま「後で具体化する」と進めた | 曖昧さに気づいた時点で具体化する。共通ルール5 Step 3C に従う。 |
| 別 feature への依存に気づいたが記録せず進めた | 依存関係を両方の feature ファイルに記録する。共通ルール5 Step 3A に従う。 |
