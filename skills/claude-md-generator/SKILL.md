# CLAUDE.md 生成・更新

## このスキルの役割

プロジェクトの CLAUDE.md を生成・更新する。CLAUDE.md は Claude Code が毎セッション開始時に読み込むファイルで、プロジェクト固有の文脈をエージェントに伝える。

## CLAUDE.md の基本原則

### 200行以下を厳守する

CLAUDE.md が長くなるほど、エージェントの指示遵守率が下がる。
「この行を削除したら Claude が間違いを犯すか？」と問い、答えが No なら削除する。

### コードから推測できることは書かない

- ESLint/Prettier の設定は設定ファイルにある → CLAUDE.md に書かない
- package.json のスクリプトは読める → 標準的なコマンドは書かない
- 既存コードのパターンは推測できる → コーディング規約は書かない

### 詳細はスキルや docs/ に委ねる（Progressive Disclosure）

CLAUDE.md には「何を参照すべきか」だけ書き、詳細は書かない。

```markdown
# ❌ 悪い例：CLAUDE.md にルールを全部書く
## テストルール
テストは必ずテスト対象の隣に置く。E2Eは tests/e2e/ に置く。
モックは外部APIのみ許可。DBのモックは禁止。
Red-Green-Refactor サイクルに従う...（50行続く）

# ✅ 良い例：参照先だけ書く
## テスト
web-testing スキルの基本理念に従う。
```

## CLAUDE.md に書くべき内容（6セクション）

### 1. プロジェクト概要（1〜3行）

何を作っているか。ユーザーは誰か。目的は何か。

```markdown
## プロジェクト概要
{プロジェクト名} — {対象ユーザー}向けの{何をするアプリ}。
{補足があれば1行}。
```

### 2. 技術的な判断・制約

コードから推測**できない**選択の理由や制約。

```markdown
## 技術的な判断
- Next.js App Router を使用（Pages Router は使わない）
- 状態管理: {Zustand / Jotai / Context 等}（選定理由: {理由}）
- UIライブラリ: shadcn/ui + Tailwind CSS + lucide-react
- {その他、コードから読めない技術判断}
```

**書かないもの：**
- `npm run dev` で起動する → package.json で分かる
- TypeScript を使う → tsconfig.json で分かる
- ESLint のルール → .eslintrc で分かる

### 3. ディレクトリ構成

採用パターン名と、一言の説明のみ。詳細は project-structure スキルに委ねる。

```markdown
## ディレクトリ構成
機能別（Feature-Based）+ レベル2一方向依存 + コロケーション。
詳細は project-structure スキルに従う。
```

### 4. docs/ 参照マップ

docs/ にあるドキュメントへの参照。エージェントが「どこを見ればよいか」を知るため。

```markdown
## ドキュメント
- 全体設計: docs/architecture.md
- 機能仕様: docs/features/（1機能 = 1ファイル。要求・要件・技術仕様・タスク。frontmatter に進捗を記録）
- 記録: docs/notes/（ブレスト、ADR、日誌、セキュリティレビュー等）
```

**存在するドキュメントのみ記載する。** 空の docs/ ディレクトリへの参照は書かない。

### 5. 外部サービス・環境変数

使用する外部サービスと、必要な環境変数名。**値は絶対に書かない。**

```markdown
## 外部サービス
- 認証: {Clerk / NextAuth / Firebase Auth 等}
- DB: {Supabase / PlanetScale / Prisma + PostgreSQL 等}
- 決済: {Stripe 等}（テスト環境あり）

## 環境変数
.env.local に以下が必要:
- DATABASE_URL
- NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY
- STRIPE_SECRET_KEY
```

### 6. 学習ガイド設定

learning-guide スキルの動作を制御する設定。ユーザーの学習スタイルに合わせて調整する。

```markdown
## 学習ガイド
explanation-level: 2
content-level: 基礎
content-scope: 広い
```

| 設定 | 値 | 意味 |
|------|---|------|
| explanation-level | 0 / 2 / 3 | 説明の量（なし / 簡潔 / 詳細+記録） |
| content-level | 基礎 / 実践 / 判断者 | どこまでかみ砕くか |
| content-scope | 狭い / 広い | 周辺の話題まで広げるか |

### 7. 禁止事項

コードからは推測できない制約。過去のミスから学んだルール等。

```markdown
## 禁止事項
- {具体的な禁止事項}
- {具体的な禁止事項}
```

## CLAUDE.md 生成手順

### 新規プロジェクトの場合

1. **プロジェクトの情報を収集する**
   - package.json を確認（技術スタック）
   - 既存の docs/ を確認（ドキュメントの有無）
   - .env.example または .env.local を確認（環境変数）
   - 既存のディレクトリ構造を確認

2. **ユーザーに確認する**
   - プロジェクト概要（何を作っているか）
   - コードから推測できない技術判断
   - 禁止事項

3. **テンプレートに沿って生成する**
   - 上記6セクションを埋める
   - 200行以下であることを確認
   - プロジェクトルートに CLAUDE.md として保存

### 既存の CLAUDE.md を更新する場合

1. **現在の CLAUDE.md を読む**
2. **変更のトリガーを確認する**（PRD追加、技術スタック変更、構成変更等）
3. **該当セクションのみ更新する**（全体を書き直さない）
4. **200行以下を維持する**（追加したら、不要な行を削除する）

## CLAUDE.md テンプレート

### テンプレートA: 通常のプロジェクト用

```markdown
# {プロジェクト名}

## プロジェクト概要
{1〜3行の説明}

## 技術的な判断
- {コードから推測できない技術判断1}
- {コードから推測できない技術判断2}

## ディレクトリ構成
機能別（Feature-Based）+ レベル2一方向依存 + コロケーション。
詳細は project-structure スキルに従う。

## ドキュメント
- 全体設計: docs/architecture.md
- 機能仕様: docs/features/（1機能 = 1ファイル。frontmatter に進捗を記録）
- 記録: docs/notes/

## 外部サービス
- {サービス名}: {用途}

## 環境変数
.env.local に以下が必要:
- {環境変数名}

## 学習ガイド
explanation-level: 2
content-level: 基礎
content-scope: 広い

## 禁止事項
- {禁止事項}
```

### テンプレートB: フォーク・自作プラグイン用

OSS をフォークして個人開発として運用する場合、または自作の Claude Code
プラグインの場合は、こちらのテンプレートを使う。**本家への PR を前提
とする Contributor Guidelines を絶対に残さない**ことが肝心。

このテンプレートは harness 自身の CLAUDE.md（このリポジトリの CLAUDE.md）の
構造を一般化したもの。

```markdown
# {プラグイン名} — 開発ガイドライン

このリポジトリは [{元リポジトリ名}]({URL})（{ライセンス}, Copyright (c) {年} {元著者}）
からのフォークで、**個人開発として運用しているプラグイン**である。
本家への PR 提出は前提にしない。

## 設計思想（変更しないこと）

このプラグインの中核となる考え方。スキルや構造を変更するときは、
これらと矛盾しないかを必ず確認する。

1. **{設計思想1}**: {1行で説明}
2. **{設計思想2}**: {1行で説明}
3. **{設計思想3}**: {1行で説明}

## スキル変更時のルール

このリポジトリ自体に対して変更を加えるときの段取り。

### agents/ の変更
- 単体で動作確認すれば OK（コミット前に1セッションで挙動を確認）
- 影響範囲が局所的なので比較的気軽に試せる

### skills/SKILL.md の変更
- スキルは「コードのようにエージェントの挙動を形成する文書」である
- 既存スキルへの追記: 1セッションで挙動確認してからコミット
- 新規スキル追加: 2-3 セッションで実際に呼び出されるか確認
- 挙動制御の核となる文言（Red Flags、Rationalization List 等）を変えるときは、
  変更前後の挙動比較を docs/notes/ に記録する

### 設計思想に関わる変更
- 上記の設計思想に影響する変更は、**実施前**に
  docs/notes/{日時}_adr-{番号}_{タイトル}.md で ADR を残す

### コミットの粒度
- 1コミット1テーマ
- メッセージは Conventional Commits 形式

## 上流（{元リポジトリ}）との関係

- **追従しない**: フォルダ構造・スキル粒度・命名規則は独自に最適化する
- **取り込みたい変更がある場合**: git remote として `upstream` を追加し、
  cherry-pick で必要なものだけ取り込む。マージはしない
- **独自スキル**: {独自に書き起こした・大幅改変したスキル名} 等は
  上流の同名ファイルで上書きしないこと

## 上流の遺物に注意

このリポジトリには上流由来のファイル・記述が残っていることがある。発見したら：

- 単純に不要なもの（古い計画ドキュメント等）→ 削除
- 内容を独自仕様に書き換えれば使えるもの → 書き換える
- 用途が不明なもの → docs/notes/ に「TODO: 上流遺物の調査」として記録

## ファイル構成

```
.
├── .claude-plugin/plugin.json
├── README.md
├── CLAUDE.md
├── agents/
├── commands/
├── hooks/
├── skills/
└── docs/notes/
```

## Pull Request について

このリポジトリへの PR は本家のものとは無関係。フォーク内でのレビューが
必要なら通常通り PR を立てるが、本家リポジトリへの逆流（pushup）はしない。
```

### テンプレート選択フロー

```
Q: このプロジェクトは何か？
  ├── 通常のアプリケーション・Web サービス → テンプレートA
  ├── 自作の Claude Code プラグイン → テンプレートB
  ├── OSS をフォークしたプラグイン → テンプレートB（必須）
  └── OSS をフォークしたアプリケーション → テンプレートA をベースに、
     上流との関係セクションをBから取り込む
```

### フォーク時の典型的な失敗パターン

| 失敗 | 結果 | 対処 |
|---|---|---|
| 元の CLAUDE.md（Contributor Guidelines）をそのまま残した | 毎セッション「PR 拒否率高い」「slop は閉じる」等の警告で動作にバイアス | テンプレートB で完全に書き換える |
| 「フォーク」とだけ書いて元との関係を明示しない | 上流の変更を取り込むべきか判断できない | 「追従しない」「cherry-pick で取り込む」を明記 |
| 独自スキル名のリストを書かなかった | 上流からマージしたとき独自分が上書きされる | 独自スキルを列挙して保護 |
| 設計思想を書かなかった | スキル追加・変更で哲学がブレる | 3-5 個の設計思想を明文化 |

## CLAUDE.md の更新トリガー

| イベント | 更新内容 |
|---------|---------|
| feature ファイルを新規作成 | docs/ 参照マップに追加 |
| 技術スタック変更（ライブラリ追加・削除） | 技術的な判断セクションを更新 |
| 外部サービス追加 | 外部サービス・環境変数セクションを更新 |
| 禁止事項を発見（バグやミスから） | 禁止事項セクションに追加 |
| ディレクトリ構成の例外を設けた | ディレクトリ構成セクションに注記 |
| ユーザーが学習ガイド設定を変更したい | 学習ガイドセクションの値を更新 |

## Red Flags — 以下に該当したら立ち止まること

| Red Flag | 正しい対応 |
|----------|-----------|
| CLAUDE.md が200行を超えた | 不要な行を削除するか、スキルや docs/ に移動する |
| コーディング規約を書こうとした | ESLint/Prettier と既存コードに任せる。書かない |
| 開発コマンド一覧を書こうとした | package.json で分かる標準的なものは書かない |
| テストルールの詳細を書こうとした | 「web-testing スキルに従う」の1行で済む |
| ディレクトリ構造の全体図を書こうとした | 「project-structure スキルに従う」の1行で済む |
| 環境変数の値を書いた | 値は絶対に書かない。変数名のみ |
| 存在しない docs/ への参照を書いた | 存在するドキュメントのみ記載する |
