# Web Testing Skill

Next.js（React）ウェブアプリケーションのテストをClaude Codeが実施するためのSkill。

**学習ガイド:** テスト種類の選択時・テスト戦略の決定時に **learning-guide** スキルを参照し、CLAUDE.md の説明レベル設定に応じてテスト概念をユーザーに説明する。説明レベルが 0 の場合はスキップする。

## 基本理念

**テストの目的 = ユーザーがやりたいことができることを確認する**

```
最も重要な質問: 「ユーザーがやりたいことができるか？」

この質問に答えるのが「受け入れテスト（E2E）」
統合テスト・単体テストは、E2Eを支える土台
```

### テストを先に書く（TDD 鉄則）

```
本番コードを書く前に、必ず失敗するテストを先に書くこと
```

これは「テストファースト」と呼ばれる原則で、すべての Phase に適用される。

テストを後から書くと、以下の問題が起きる：

- テストが最初から成功してしまい、「本当にバグを検出できるテスト」なのか確認できない
- 実装に引きずられて、実装の動作を追認するだけのテストになる
- エッジケース（境界値や異常入力）を見落としやすい

テストを先に書くと：

- テストが失敗するのを確認してから実装するので、テストの検出力が保証される
- 「何をすべきか」を先に定義してから「どう作るか」に進むので、設計がクリーンになる

#### Red-Green-Refactor サイクル

すべてのテスト作成は、以下の 3 ステップで行う：

1. **RED（赤）** — まだ実装していない機能のテストを1つ書く → テストを実行 → **失敗することを確認**
2. **GREEN（緑）** — テストを通す最小限のコードだけを書く → テストを実行 → **成功することを確認**
3. **REFACTOR（整理）** — テストが緑のまま、コードを整理する（名前の改善、重複の除去など）

**重要：各ステップで必ずテストを実行して結果を確認すること。**

#### 実装コードを先に書いてしまった場合

**その実装コードを削除して、テストから書き直すこと。**

- 「参考として残す」は禁止 — 見ながら書くと、テスト後付けと同じ結果になる
- 「もったいない」は錯覚 — 検証されていないコードを残す方がリスクが高い
- 「手動で確認済み」は不十分 — 手動テストは記録に残らず、変更のたびに再実行できない

### Red Flags — 以下に該当したら立ち止まること

以下のいずれかに当てはまる場合、やり方を間違えている可能性が高い。
作業を止めて、見直すこと。

| Red Flag | 正しい対応 |
|----------|----------|
| テストより先に実装コードを書いた | 実装を削除。テストから書き直す |
| テストが最初から成功した | テストが間違っている。失敗するテストに修正 |
| なぜテストが失敗したか説明できない | テストか実装の理解が不足。原因を特定してから進む |
| テストの大半がモックの検証になっている | モック過多。実コードのテストに書き換える |
| DB をモックしている | 実 DB でテストする。Docker でテスト用 DB を起動 |
| 内部 API Route をモックしている | 実際のルートハンドラを通すテストに書き換える |
| 「モックしないとテストできない」と思った | 設計の結合度が高すぎる。依存性注入を検討 |
| 「テスト後付けでも同じ」と思った | 同じではない。テスト後付けは実装への追認になる |
| 「手動で確認したからテスト不要」と思った | 手動テストは記録に残らない。自動テストを書く |
| 「シンプルすぎてテスト不要」と思った | シンプルなコードも壊れる。テストは30秒で書ける |
| 「今回だけスキップ」と思った | 例外はない。スキップは習慣になる |
| 「TDDは教条的、実用的に行こう」と思った | TDD は実用的。テスト後付けデバッグの方が遅い |

**上記すべてに共通する対処：作業を止めて、テストから書き直す。**

## ★最重要：テストの優先順位

```
受け入れテスト（E2E）= ゴール
    │
    │  「ユーザーがやりたいこと」ができるか確認
    │
    │  これが通るためには...
    │
    ├─→ 統合テスト が通っている必要がある
    │       │
    │       │  API + DB連携が正しい
    │       │
    │       └─→ 単体テスト が通っている必要がある
    │               │
    │               │  個々のロジックが正しい
    │
    └───────────────┘

E2Eが最上位の目標（ゴール）
統合テスト・単体テストはその土台（前提条件）
土台がなければE2Eは成り立たない
```

## テスト作成の流れ

```
Phase 0: 環境確認（★必ず最初に行う）
    │
    ├─ テスト実行に必要な環境が整っているか確認
    ├─ 不足があれば → ユーザーに設定手順を案内
    └─ 設定完了を確認してから次へ進む
    
Phase 1: E2Eテストの設計
    │
    ├─ ユースケースを洗い出す
    ├─ 正常系・異常系・エッジケースを列挙
    ├─ ビジュアル確認ポイントを特定
    └─ セキュリティ確認ポイントを特定

Phase 2: 土台を作る（単体テスト・統合テスト）
    │
    ├─ 単体テスト: ロジックの正確性を検証
    ├─ 統合テスト: API・DB連携を検証
    └─ モックは最小限に（実DBを使う統合テストを重視）

Phase 3: E2Eテストを実装・実行
    │
    ├─ 機能テスト: 操作 → 結果確認
    ├─ ビジュアルテスト: UI・レイアウト確認
    └─ セキュリティテスト: 認証・認可・不正入力

Phase 4: パフォーマンステスト
    │
    ├─ 応答時間の確認
    └─ 負荷テスト（必要に応じて）

Phase 5: 網羅性の確認・報告
    │
    ├─ 機能カバレッジ: 全機能にE2Eがあるか
    └─ コードカバレッジ: 参考値として確認
```

## Phase 0: 環境確認（★必ず最初に行う）

**テスト環境が整っていなければ、テストを書き始めない。**

### 確認すべき項目

1. **テストDB**
   - テスト用のデータベースが用意されているか
   - マイグレーションは完了しているか
   - シードデータは投入されているか

2. **環境変数**
   - `.env.test` または `.env.local` が設定されているか
   - 必要なAPI Keyが設定されているか

3. **外部サービス**
   - OAuth設定（テストユーザーの追加等）
   - 外部API（サンドボックス/テスト環境）
   - ストレージ（S3/R2等のテストバケット）

4. **テストツール**
   - Vitest/Playwright等がインストールされているか
   - ブラウザ（Playwright用）がインストールされているか

### 不足がある場合の対応

```markdown
## テスト実行に必要な設定

以下の設定が不足しているため、テストを開始できません。
設定完了後にお知らせください。

### 1. 環境変数

`.env.test` に以下を追加してください：

| 変数名 | 用途 | 取得方法 |
|--------|------|----------|
| `DATABASE_URL` | テストDB接続 | [取得手順を記載] |
| `OPENAI_API_KEY` | AI機能テスト | [取得手順を記載] |

### 2. テストDBのセットアップ

```bash
npm run db:push
npm run db:seed:test
```

### 3. 外部サービスの設定

[具体的な手順を記載]

---

上記の設定が完了したらお知らせください。
```

**→ 詳細は @docs/environment-setup.md を参照**

## Phase 1: E2Eテストの設計

**まず「何を確認すべきか」を明確にする。**

### ユースケースの洗い出し

各機能について、以下を列挙する：

| 分類 | 説明 | 例 |
|------|------|-----|
| **正常系** | 想定通りの操作 | 有効なデータで登録成功 |
| **異常系** | エラーになるべき操作 | 無効なデータでエラー表示 |
| **エッジケース** | 境界値・特殊な状況 | 最大文字数、空入力 |
| **準正常系** | エラーだが回復可能 | セッション切れ→再ログイン |

### 確認ポイントの特定

| 観点 | 確認すること |
|------|-------------|
| **機能** | 操作の結果が正しいか（DBに保存、画面遷移等） |
| **ビジュアル** | UIが意図通りに表示されるか |
| **セキュリティ** | 認証・認可・入力検証が正しいか |
| **パフォーマンス** | 応答時間が許容範囲内か |

**→ 詳細は @docs/e2e-testing.md を参照**


## モック許可制度

モック（偽物の代替品）は、テスト対象以外の部分を簡略化するための仕組みだが、
使いすぎると「偽物同士のやりとりをテストしているだけ」になり、本物の動作を検証できなくなる。

そのため、モックの使用には以下のルールを適用する：

### モック使用許可テーブル

| 対象 | モック使用 | 理由 |
|------|-----------|------|
| 外部 API（Stripe, SendGrid 等） | ✅ 許可 | テスト環境から本番APIを呼ぶのは危険・高コスト |
| データベース | ❌ 禁止 | 実DBで統合テストすべき。Docker等でテスト用DB起動 |
| 内部 API Route（Next.js等） | ❌ 禁止 | 実際のルートハンドラを通してテストすべき |
| ブラウザ操作（Playwright） | ❌ 禁止 | 実ブラウザでE2Eテストすべき |
| ファイルシステム | ⚠️ 状況による | テスト用の一時ディレクトリを使うのが望ましい |
| 時間（Date, Timer） | ✅ 許可 | 時間依存テストの再現性確保のため |
| 乱数 | ✅ 許可 | 再現性確保のため |

### モック使用時の必須ルール

1. **テストファイルの先頭にモック理由を明記する**

```typescript
/**
 * モック対象: Stripe API
 * モック理由: テスト環境から本番決済APIを呼べないため
 * 代替案の検討: Stripe のテストモードも利用可能だが、CI速度のためモック採用
 */
```

2. **モックは呼び出し回数と引数も検証する**

```typescript
// ✅ 正しい：引数と回数を検証
expect(mockStripe.charges.create).toHaveBeenCalledWith({
  amount: 1000,
  currency: 'jpy',
});
expect(mockStripe.charges.create).toHaveBeenCalledTimes(1);

// ❌ 禁止：呼ばれたかだけの確認
expect(mockStripe.charges.create).toHaveBeenCalled();
```

3. **モック許可テーブルに「❌ 禁止」の対象をモックする場合は、テストケース単位でコメントに理由を書き、人間パートナーの承認を得ること**


## Phase 2: 土台を作る

**E2Eを支える単体テスト・統合テストを作成する。**

### 単体テスト

- 複雑なロジック（計算、変換、バリデーション）
- ユーティリティ関数
- カスタムフック

**→ 詳細は @docs/unit-testing.md を参照**

### 統合テスト

- API Route（リクエスト→レスポンス）
- DB操作（CRUD）
- 外部サービス連携

**重要: モックは最小限に。実DBを使う統合テストを重視する。**

**→ 詳細は @docs/integration-testing.md を参照**

## Phase 3: E2Eテストを実装・実行

### 機能テスト

**実際のブラウザで操作し、結果を確認する。**

```typescript
test('ユーザーが新規登録できる', async ({ page }) => {
  // 操作
  await page.goto('/register');
  await page.fill('[name="email"]', 'test@example.com');
  await page.fill('[name="password"]', 'SecurePass123');
  await page.click('button[type="submit"]');

  // 結果確認（UI）
  await expect(page).toHaveURL('/dashboard');
  await expect(page.getByText('登録完了')).toBeVisible();

  // 結果確認（DB）- 実際にデータが保存されたか
  const user = await db.user.findUnique({ where: { email: 'test@example.com' } });
  expect(user).not.toBeNull();
});
```

**→ 詳細は @docs/e2e-testing.md を参照**

## 意味のあるテストを書く

**テストは「常にパスする」ものではなく「失敗すべき時に失敗する」もの。**

### 禁止パターン

```typescript
// ❌ 禁止：成功してもエラーでもパスする
it('データを取得する', async () => {
  try {
    const result = await fetchData();
    expect(result).toBeDefined();
  } catch (error) {
    expect(error).toBeDefined();  // 何も検証していない
  }
});

// ❌ 禁止：アサーションが弱い
it('ユーザーを取得する', async () => {
  const user = await getUser(1);
  expect(user).toBeDefined();  // 中身が間違っていても検出できない
});
```

### 正しいパターン

```typescript
// ✅ 正常系：成功を期待し、具体的な値を検証
it('有効なIDでユーザーを取得できる', async () => {
  const user = await getUser(1);
  expect(user.id).toBe(1);
  expect(user.name).toBe('山田太郎');
});

// ✅ 異常系：エラーを期待
it('無効なIDでエラーになる', async () => {
  await expect(getUser(-1)).rejects.toThrow('User not found');
});
```

## 禁止事項

1. **Phase 0をスキップしない**
   - 環境が整っていない状態でテストを書き始めない

2. **E2Eを後回しにしない**
   - 単体テストの量産に走らない
   - まずE2Eで「ゴール」を確認する

3. **カバレッジ%を目標にしない**
   - 機能カバレッジを優先する
   - カバレッジ%は参考値

4. **意味のないテストを書かない**
   - try-catchで両方OKにしない
   - 具体的な値を検証する

5. **モックに頼りすぎない**
   - 実DBを使う統合テストを重視
   - モックは外部APIなど必要最小限に
