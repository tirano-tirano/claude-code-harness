# Claude Code テスト指針ガイド

Claude Codeが自律的にテストを作成・実行・修正する際のガイドライン。

## ★最重要：正しい優先順位でテストを書く

```
❌ 間違った進め方:
   単体テストを大量に書く → カバレッジ%を報告 → E2Eは後回し

✅ 正しい進め方:
   Phase 0: 環境確認
   Phase 1: E2Eテストの設計（何を確認すべきか明確にする）
   Phase 2: 土台を作る（単体テスト・統合テスト）
   Phase 3: E2Eテストを実装・実行
   Phase 4: パフォーマンステスト
   Phase 5: 網羅性の確認・報告
```

## Phase 0: 環境確認（★スキップしない）

**テストを書き始める前に、必ず環境を確認する。**

### 確認すべき項目

1. テストツールがインストールされているか
2. テストDBが用意されているか
3. 環境変数が設定されているか
4. 外部サービスの設定が完了しているか

### 不足がある場合

**テストを書き始めず、ユーザーに設定手順を案内する。**

```markdown
## テスト実行に必要な設定

以下の設定が不足しているため、テストを開始できません。

### 不足している設定

| 項目 | 状態 | 必要な対応 |
|------|------|-----------|
| テストDB | ❌ 未設定 | ローカルSupabaseの起動 |
| OPENAI_API_KEY | ❌ 未設定 | .env.testに追加 |

### 設定手順

1. ローカルSupabaseを起動:
   ```bash
   npx supabase start
   ```

2. .env.testに環境変数を追加:
   ```
   OPENAI_API_KEY=sk-...
   ```

---

設定が完了したらお知らせください。
```

## Phase 1: E2Eテストの設計

**まず「何を確認すべきか」を明確にする。**

### やること

1. 各機能のユースケースを洗い出す
2. 正常系・異常系・エッジケースを列挙する
3. ビジュアル・セキュリティの確認ポイントを特定する

### 成果物の例

```markdown
## 機能: ユーザー登録

### E2Eテストケース

| # | 種別 | テストケース | 確認内容 |
|---|------|-------------|----------|
| 1 | 正常系 | 有効なデータで登録 | DB保存、リダイレクト |
| 2 | 異常系 | 無効なメール | エラー表示、DB未保存 |
| 3 | 異常系 | 重複メール | エラー表示、DB未保存 |
| 4 | エッジ | 最大文字数 | 正常登録 |
| 5 | セキュリティ | XSSペイロード | エスケープ |
| 6 | ビジュアル | フォーム表示 | レイアウト確認 |
```

## Phase 2: 土台を作る

**E2Eを支える単体テスト・統合テストを作成する。**

### 重要: モックは最小限に

```typescript
// ❌ 悪い例：DBをモック
vi.mock('@/lib/db');

// ✅ 良い例：実DBを使い、外部APIだけモック
import { db } from '@/lib/db';  // 実DB
vi.mock('@/lib/email');  // 外部サービスはモック
```

### 単体テストの対象

- バリデーションロジック
- 計算・変換ロジック
- ユーティリティ関数

### 統合テストの対象

- API Route（リクエスト→レスポンス）
- DB操作（CRUD）

## Phase 3: E2Eテストを実装・実行

**Phase 1で設計したテストケースを実装する。**

### E2Eテストの書き方

```typescript
test('ユーザーが新規登録できる', async ({ page }) => {
  // 1. 操作
  await page.goto('/register');
  await page.fill('[name="email"]', 'test@example.com');
  await page.fill('[name="password"]', 'SecurePass123!');
  await page.click('button[type="submit"]');

  // 2. UI上の結果を確認
  await expect(page).toHaveURL('/dashboard');

  // 3. DBの変更を確認（★重要）
  const user = await db.user.findUnique({ 
    where: { email: 'test@example.com' } 
  });
  expect(user).not.toBeNull();
});
```

## 意味のあるテストを書く

### 禁止パターン

```typescript
// ❌ 禁止：try-catchで両方OKにする
it('データを取得する', async () => {
  try {
    const result = await fetchData();
    expect(result).toBeDefined();
  } catch (error) {
    expect(error).toBeDefined();  // 何も検証していない
  }
});

// ❌ 禁止：アサーションが弱い
it('ユーザーを取得する', async () => {
  const user = await getUser(1);
  expect(user).toBeDefined();  // 中身は？
});

// ❌ 禁止：テストを緩くして通す
// 失敗するテストの期待値を緩くするのは禁止
// 実装を修正すること
```

### 正しいパターン

```typescript
// ✅ 具体的な値を検証
it('ユーザーを取得する', async () => {
  const user = await getUser(1);
  expect(user.id).toBe(1);
  expect(user.name).toBe('山田太郎');
});

// ✅ 異常系は別のテストで
it('存在しないIDでエラー', async () => {
  await expect(getUser(999)).rejects.toThrow('Not found');
});
```

## テスト失敗時の対処

### ✅ 正しい対処

1. エラーメッセージを確認
2. 原因を特定（実装のバグ？テストの間違い？）
3. **実装を修正する**
4. 再テスト

### ❌ 禁止されている対処

1. テストの期待値を緩くする
2. try-catchを追加してエラーを握りつぶす
3. テストをスキップする

## 自動修正のガイドライン

### 修正してよいケース

| ケース | 例 |
|--------|-----|
| 実装のバグ | off-by-oneエラー、nullチェック漏れ |
| テストコードの問題 | セレクタの間違い、待機処理の不足 |
| 軽微なミス | タイポ、インポート漏れ |

### 修正前に確認が必要なケース

| ケース | 対応 |
|--------|------|
| 仕様の問題 | ユーザーに確認 |
| 設計の問題 | ユーザーに確認 |
| セキュリティの問題 | ユーザーに確認 |

### 絶対にやってはいけないこと

| 禁止事項 | 理由 |
|----------|------|
| テストを緩くして通す | バグを見逃す |
| try-catchを追加 | テストの意味がなくなる |
| テストをスキップ | カバレッジが下がる |
