# 学習ガイド（Learning Guide）

## このスキルの役割

開発作業中に、AIが行っている判断・選択の背景をユーザーに説明する。
ユーザーが「コードは書かないが、仕組みと判断の理由を理解している」状態を目指す。

**このスキルは2つの方法で動作する：**

1. **自動モード** — 他のスキル（development-cycle 等）の作業中に、設定に応じて概念を説明する
2. **手動モード** — ユーザーが「なぜこうする？」「これは何？」と聞いたときに発動する
3. **まとめモード** — 「〇〇について学んだことをまとめて」で過去の学習記録を整理する

## 初回チェック（CLAUDE.md への自動追記）

このスキルが参照されたとき、CLAUDE.md に「学習ガイド」セクションが存在するか確認する。
存在しない場合、以下のデフォルト設定を CLAUDE.md に追記する：

```markdown
## 学習ガイド
explanation-level: 2
content-level: 基礎
content-scope: 広い
```

追記後、ユーザーに「学習ガイドの設定を CLAUDE.md に追加しました。必要に応じて値を変更してください。」と伝える。

## 設定

CLAUDE.md に以下を記載する。セッション中に口頭で変更可能。次のタスク/ステップに移ったらデフォルトに戻る。

```
explanation-level: 2
content-level: 基礎
content-scope: 広い
```

### 説明レベル（explanation-level）

AIがどの程度説明するかを制御する。

| レベル | 動作 |
|-------|------|
| 0 | 説明なし。作業のみ行う |
| 2 | 簡潔な説明（2〜3行）。作業の流れを止めない |
| 3 | 詳細な説明 + 学習記録をファイルに保存する |

### 内容レベル（content-level）

ユーザーの前提知識に合わせて、説明のかみ砕き度を制御する。

| レベル | 前提 | 説明の仕方 |
|-------|------|-----------|
| 基礎 | 技術知識がほぼない | そもそもそれが何なのかから説明する。比喩や身近な例を使う。「データベースは情報を整理して保存する倉庫のようなもの」レベル |
| 実践 | 基本概念は分かっている | 「なぜこの選択をしたか」「別の選択肢と比べてどうか」「放置するとどんなリスクがあるか」にフォーカスする |
| 判断者 | 実践的な理解がある | 「この選択が将来のコスト・スケーラビリティにどう影響するか」「業界の主流パターンとの比較」「どういう条件なら別の選択をすべきか」を説明する |

### 内容スコープ（content-scope）

説明の範囲を制御する。

| スコープ | 動作 |
|---------|------|
| 狭い | 今の作業に直結する概念だけ説明する |
| 広い | 関連するセキュリティ、コスト、スケーラビリティ、リスク、代替案まで広げて説明する |

### 設定の組み合わせ例

| 設定 | 場面の例 | 説明内容 |
|------|---------|---------|
| レベル2 × 基礎 × 狭い | API設計時 | 「APIとは、アプリの中で『こういうデータをください』と頼む窓口のことです。今、ユーザー情報を取得する窓口を作っています」 |
| レベル2 × 基礎 × 広い | API設計時 | 上記に加えて「この窓口には鍵をかけないと、誰でもユーザー情報を見られてしまいます（認証）。また、大勢が同時に窓口に来ると処理が追いつかなくなります（スケーラビリティ）」 |
| レベル2 × 実践 × 狭い | API設計時 | 「RESTful設計でGETメソッドを使っています。パスにIDを含めるのは、リソースを一意に特定するためです」 |
| レベル3 × 実践 × 広い | API設計時 | 上記 + 「REST vs GraphQL のトレードオフ」「レート制限の必要性」「APIバージョニング戦略」を詳しく説明し、学習記録に保存 |

## 説明の対象トピック

以下のトピックについて、作業中に該当する場面で説明する。

### アーキテクチャ

- なぜファイルを分けるのか（関心の分離）
- なぜ依存方向を一方向にするのか
- レイヤー構造の考え方（app/ → features/ → entities/ → lib/）
- モノリスとマイクロサービスの違い

### 技術仕様 — API設計

- API（アプリケーションプログラミングインターフェース）とは何か
- RESTful設計の考え方（リソース、HTTPメソッド、ステータスコード）
- リクエストとレスポンスの構造
- 認証付きAPIと公開APIの違い
- APIバージョニング（なぜバージョンが必要か）
- レート制限（なぜ制限するのか）
- エラーレスポンスの設計

### 技術仕様 — DB設計

- データベースとは何か（データを構造的に保存する仕組み）
- テーブル、カラム、行の関係
- リレーション（テーブル同士の関係: 1対1、1対多、多対多）
- 正規化（なぜデータを分けて持つのか）と非正規化（なぜあえてまとめるのか）
- インデックス（なぜ検索が速くなるのか）
- マイグレーション（DBの構造を安全に変更する手順）
- バックアップとリストア

### 技術仕様 — UI設計

- コンポーネント指向（画面をパーツに分けて組み合わせる考え方）
- レスポンシブデザイン（画面サイズに応じてレイアウトを変える）
- 状態管理（画面の中でデータがどう流れるか）
- アクセシビリティ（誰でも使えるUIにする考え方）
- デザインシステム（共通のUI部品を統一するルール）

### セキュリティ

- 認証（あなたは誰か）と認可（あなたは何ができるか）の違い
- 入力値を信用してはいけない理由（インジェクション攻撃）
- HTTPS（通信の暗号化）がなぜ必要か
- セッション管理とトークン
- CORS（異なるドメイン間のアクセス制御）
- 環境変数とシークレット管理

### テスト

- なぜテストを書くのか（変更しても壊れていないことを確認する）
- 単体テスト、統合テスト、E2Eテストの違い
- テスト駆動開発（TDD）の考え方（なぜテストを先に書くのか）
- モックとスタブ（なぜ偽物を使うのか、いつ本物を使うべきか）
- カバレッジ（テストがどの程度コードを網羅しているか）

### コスト・スケーラビリティ

- 外部サービスの料金体系（従量課金、固定料金）
- ユーザー数が増えたときに何が変わるか
- キャッシュ（同じ計算を繰り返さない工夫）
- 非同期処理（重い処理を後回しにする考え方）
- 運用コスト（サーバー代、監視、メンテナンス）

### 判断・選択

- 技術選定の考え方（なぜこのフレームワーク/ライブラリを選んだか）
- 「今は便利だが将来困る」パターンの見分け方
- 技術的負債とは何か（後回しにしたツケ）
- ビルド vs バイ（自分で作るか、既製品を使うか）

## 自動モードの動作

他のスキルの作業中に、以下のタイミングで説明を挿入する。

### 説明を挿入するタイミング

| 呼び出し元スキル | 説明タイミング |
|----------------|--------------|
| development-cycle | 各ステップの開始時（設計、実装、テスト等） |
| security-review | 各チェックカテゴリの実行時 |
| web-testing | テスト種類の選択時、テスト戦略の決定時 |
| project-structure | ファイル配置の判断時、依存ルールの適用時 |
| brainstorming-with-docs | 設計判断の場面、選択肢の比較時 |
| ui-design | UI設計パターンの選択時、コンポーネント設計時 |

### 説明の形式

**説明レベル 2（簡潔）の場合：**

作業の出力の中に、自然な形で2〜3行の説明を挿入する。作業の流れを止めない。

```
（ここから作業出力）
API エンドポイント POST /api/users を設計します。

【なぜこうするか】POST メソッドを使うのは「新しいデータを作成する」操作だからです。
データを取得するときは GET、更新するときは PUT、削除するときは DELETE を使います。
これが RESTful 設計の基本ルールです。

（作業出力の続き）
```

**説明レベル 3（詳細+記録）の場合：**

レベル 2 の内容に加えて、背景・代替案・リスクを詳しく説明する。
説明内容を学習記録ファイルに保存する。

## 手動モードの動作

ユーザーが明示的に質問したとき（「なぜ？」「これは何？」「説明して」等）は、説明レベルの設定に関わらずレベル 3 相当で回答する。ただし学習記録の保存は、説明レベルが 3 のときのみ行う。

## まとめモードの動作

ユーザーが「〇〇について学んだことをまとめて」と言ったとき：

1. `docs/notes/` の `_learning_` を含むファイルを全て読む
2. 指定されたテーマに関連する内容を抽出する
3. 重複を整理し、理解しやすい順序に並べ替える
4. 会話の中で表示する（デフォルト）
5. 「保存して」と言われたら `docs/notes/{日時}_learning-summary_{テーマ}.md` に保存する

まとめの構成：

```markdown
# {テーマ} — 学習まとめ

## このテーマの全体像
（テーマの位置づけ、なぜ重要か）

## 学んだこと
（時系列ではなく、理解しやすい順序で整理）

## 判断のポイント
（このテーマに関連する意思決定で押さえるべき点）

## まだ触れていないこと
（このテーマで今後学ぶ余地がある領域）
```

## 学習記録のファイル形式

説明レベル 3 のとき、説明した内容を以下の形式で保存する。

**ファイル名:** `docs/notes/{日時}_learning_{テーマ}.md`

**形式:**

```markdown
# {テーマ}

作業: {何をしていたか（例: F-001 のAPI設計）}

## 概念

### {概念名1}
{説明内容}

### {概念名2}
{説明内容}

## 判断とその理由
- {選択した内容}: {なぜそうしたか}
- 代替案: {他に何があったか、なぜ選ばなかったか}

## 関連するリスク・注意点
- {リスクや将来の制約}
```

## 設定のリセットルール

- セッション中に口頭で設定を変更できる（「詳しく教えて」→ レベル3、「解説いらない」→ レベル0）
- **次のタスクまたはステップに移ったら、CLAUDE.md のデフォルト値に自動的に戻る**
- 「ステップの切り替わり」は以下を指す:
  - development-cycle の Step が変わるとき
  - 別の feature ファイルに移るとき
  - 新しいスキルが発動するとき

## Red Flags — 以下に該当したら立ち止まること

| Red Flag | 正しい対応 |
|----------|-----------|
| 説明レベル 0 なのに説明を挿入した | 説明を省略する。作業のみ行う |
| 専門用語を説明なしに使った（内容レベル「基礎」のとき） | 平易な言葉に言い換え、簡単な例を添える |
| 作業の流れが説明で止まった（説明レベル 2 のとき） | 説明を2〜3行に収める。詳細はレベル 3 で |
| 学習記録を保存し忘れた（説明レベル 3 のとき） | 説明した内容を docs/notes/ に保存する |
| ユーザーが「なぜ？」と聞いたのに説明レベル 0 だから無視した | 手動の質問には常にレベル 3 相当で回答する |
| 同じ概念を毎回最初から説明した | 過去の学習記録を確認し、既知の部分は省略するか参照を示す |
