# 事故からの復旧（Recovering From Mistake）

## このスキルの役割

harness は「正しい順序で進めれば正しい結果」を前提にしているが、
現実には事故や違反が起きる。本スキルは **事故からどう戻すか** の
手順集。

**基本原則：**
- パニックで広範囲に手を加えない
- 復旧の前に**現状を保存**する（git stash / branch backup）
- 「やり直す前に、何が起きたかを書く」

## ケース別の復旧手順

事故のパターンごとに、Step を示す。判定できないときは
「Section 0: まず現状を保存する」だけやって、ユーザーに状況を伝える。

---

## Section 0: まず現状を保存する（全ケース共通）

復旧作業に着手する前に、必ず以下を実行する：

```bash
# 現在のブランチ名を確認
git branch --show-current

# 未コミットの変更があれば一旦退避（あとで戻せるように）
git stash push -u -m "rescue-$(date +%Y%m%d-%H%M%S)"

# 現在の HEAD をバックアップブランチで残す
git branch backup/$(date +%Y%m%d-%H%M%S)

# 直近のコミットを確認
git log --oneline -10
```

これだけやっておけば、**復旧で何を試しても元に戻せる**。
事故対応は焦りやすいが、ここを省略するとさらなる事故に繋がる。

## Section 1: TDD の順序を間違えた（実装を先に書いた）

### 症状
- テストを書く前に本番コードを書いてしまった
- 「とりあえず動かしてからテストを書こう」と思った

### 復旧手順

1. **書いてしまった本番コードを削除する**
   ```bash
   # 該当ファイルを git の最後の状態に戻す
   git restore {ファイルパス}

   # または手動で該当部分を削除
   ```

2. **テストから書き直す**
   - development-cycle の TDD 鉄則に従う
   - Red → Green → Refactor の順を守る

3. **「参考として残す」は禁止**
   - 削除した実装コードを「コピーしておいて、後で参考にする」のは
     テスト後付けと同じ結果になる
   - 検証されていないコードを残す方がリスクが高い

### なぜこれが必要か

実装を見ながらテストを書くと、テストが「実装を再描画するだけ」になる。
本来の TDD は「期待する振る舞いを先に定義し、それに合わせて実装を作る」
という順序が大事。

---

## Section 2: 完了済みタスクのチェックを誤って外した

### 症状
- feature ファイルの `- [x]` を `- [ ]` に戻してしまった
- progress カウントが狂った

### 復旧手順

1. **git で元に戻す**
   ```bash
   # 直近のコミットからチェックボックス状態を復元
   git diff HEAD docs/features/F-xxx_*.md
   git restore docs/features/F-xxx_*.md
   ```

2. **コミット済みの変更だった場合は手動修正**
   - チェックボックスを `- [x]` に戻す
   - frontmatter の progress を実態に合わせる

3. **なぜチェックを外したかを思い出す**
   - 「やり直しが必要」と思った場合 → development-cycle 共通ルール5
     Step 3B に従い、**修正タスクを末尾に追記する**（既存チェックは外さない）
   - 単なる誤操作 → 元に戻すだけで OK

### なぜこれが必要か

チェックボックスは「過去にやった事実」の記録。外すと「やり直し」
として扱われ、Ralph Loop が再度処理してしまう。修正が必要なら
別タスクとして追記するのが正しい運用。

---

## Section 3: feature ファイルの progress と実態がずれた

### 症状
- frontmatter の `progress: 5/12` だが、実際のチェックボックスを数えると違う
- status が in-progress のまま、実態は done に近い（もしくは逆）

### 復旧手順

1. **チェックボックスを数え直す**
   ```bash
   # 完了済みタスク数
   grep -c "^- \[x\]" docs/features/F-xxx_*.md

   # 全タスク数
   grep -cE "^- \[(x| )\]" docs/features/F-xxx_*.md
   ```

2. **frontmatter を実態に合わせる**
   ```yaml
   progress: {完了数}/{全タスク数}
   status: {実態に合わせる: draft / spec / ready / in-progress / done}
   ```

3. **乖離が大きい場合は原因を調べる**
   - 仕様変更でタスクを追加し忘れた → 共通ルール5 Step 3B に従う
   - チェックを付け忘れた → 完了タスクに `[x]` を付ける
   - 仕様変更でタスクが消滅した → 該当行ごと削除し、変更履歴に記録

## Section 4: コミットを取り消したい

### 症状
- 直前のコミットを取り消したい（変更内容は残したい）
- 直前のコミットの内容自体を捨てたい
- 数コミット前の状態に戻したい

### 復旧手順

#### 4-1: 直前のコミットを取り消すが、変更内容は残したい
```bash
# コミットだけ取り消し、変更はステージングに残る
git reset --soft HEAD~1
```

#### 4-2: 直前のコミットを取り消し、変更も完全に捨てる
```bash
# 危険: 未コミットの変更も消える
# Section 0 のバックアップを取った前提で
git reset --hard HEAD~1
```

#### 4-3: 数コミット前まで戻す
```bash
# まず戻す先のコミットを確認
git log --oneline -10

# その SHA に戻す（破棄）
git reset --hard {SHA}

# または、新しいコミットとして「取り消し」を記録（履歴を残す）
git revert {取り消したい SHA}
```

### 判定基準

| 状況 | 推奨方法 |
|---|---|
| ローカルだけ、まだ push していない | reset で OK |
| 既に push 済み、自分しか触らないブランチ | reset + force push |
| 既に push 済み、他人もいるブランチ | revert（履歴を残す） |
| main / master ブランチ | revert 一択（force push 禁止） |

## Section 5: マージ・リベースで事故った

### 症状
- マージコンフリクトを誤って解消した
- リベース中におかしな状態になった
- ブランチが想定外の状態になった

### 復旧手順

1. **作業を止めて状態を確認**
   ```bash
   git status
   git log --oneline -10 --all --graph
   ```

2. **進行中のマージ/リベースを中止**
   ```bash
   # マージ中の場合
   git merge --abort

   # リベース中の場合
   git rebase --abort

   # チェリーピック中の場合
   git cherry-pick --abort
   ```

3. **それでも壊れている場合は reflog で復旧**
   ```bash
   # 過去の HEAD 履歴を確認（git reset 等の操作も含む）
   git reflog

   # 戻したい状態の HEAD@{N} を見つける
   git reset --hard HEAD@{5}
   ```

### なぜ reflog が効くか

git は HEAD の移動を 30 日保持する。`git reset --hard` で破棄したように
見える状態も、reflog 経由で取り戻せる。**「取り返しがつかない」と思っても
30 日以内なら救える**ことが多い。

## Section 6: 誤ってファイルを削除した

### 症状
- `git rm` で削除してコミットした
- `rm` でローカルから消した
- ディレクトリごと消えた

### 復旧手順

#### 6-1: コミット前ならただの restore
```bash
# ステージングに上がっているなら
git restore --staged {ファイルパス}
git restore {ファイルパス}

# 単に消しただけなら
git checkout HEAD -- {ファイルパス}
```

#### 6-2: コミット済みなら過去から取得
```bash
# どのコミットで削除されたか特定
git log --all --diff-filter=D --summary | grep -B1 {ファイル名}

# その削除コミットの直前から復元
git checkout {削除コミット}^ -- {ファイルパス}
```

---

## Section 7: ブランチを誤って削除した

### 症状
- `git branch -D` で削除してしまった
- 削除後にまだ作業していなかった大事なコミットがあった

### 復旧手順

```bash
# 削除されたブランチの最後の HEAD を reflog から探す
git reflog | grep {ブランチ名}

# その SHA からブランチを再作成
git branch {ブランチ名} {SHA}
```

reflog が消えていなければ、**削除したブランチは 30 日以内なら復元可能**。

---

## Section 8: 「全部やり直したい」と思ったとき

### 注意

「全部やり直したい」は危険な思考。多くの場合、Section 0 のバックアップを
取った上で、**部分的な修正で十分**なケースがほとんど。

### それでもリセットしたい場合

1. **Section 0 を絶対に実行する**（バックアップなしの破棄は禁止）

2. **何をリセットするか明確にする**
   ```
   Q: リセット範囲は？
     ├── 直近のコミット数件 → Section 4 に戻る
     ├── ブランチ全体 → 新ブランチを切ってやり直す
     └── プロジェクト全体 → ユーザーに確認する（このスキルでは扱わない）
   ```

3. **失敗から学ぶ**
   - 復旧後、`retrospective` スキルで「なぜ全部やり直したくなったか」を
     振り返る
   - 多くの場合、ドキュメント駆動・TDD のどこかをスキップしている

## 共通の Red Flags

| Red Flag | 正しい対応 |
|---|---|
| バックアップなしで `git reset --hard` を打とうとした | まず Section 0 を実行する |
| `--force` push を main/master にしようとした | やめる。revert に切り替える |
| 復旧で生じた疑問をユーザーに聞かずに進めた | 不明点は確認する。事故の二次災害を防ぐ |
| 「もう reflog では戻せない」と諦めた | 30 日以内なら高確率で戻せる。`git reflog` を打つ |
| 復旧の途中で別の事故対応も同時にやろうとした | 1 つずつ順番に。並行は混乱を招く |

## 復旧後にやるべきこと

事故から復旧したあと、必ず以下を行う：

1. **何が起きたかを記録する**
   - `docs/notes/{日時}_journal.md` に時系列で書く
   - または `retrospective` スキルで KPT 形式で残す

2. **同じ事故を防ぐ仕組みを考える**
   - スキルや CLAUDE.md に追記すべきルールはないか
   - チェックリスト（verification-before-completion 等）に項目を足せないか

3. **バックアップブランチを掃除する**
   - 復旧が完了し、新コミットが安定してから
   - `git branch -D backup/{日時}` で削除

## 関連スキル

- `using-git-worktrees`: worktree を使えば「壊れた worktree を捨てて作り直す」
  という形で実質的に Section 8 が安全に行える
- `git-conventions`: コミットメッセージ規約とコンフリクト解消の通常手順
- `verification-before-completion`: 事故を未然に防ぐためのチェック
- `retrospective`: 復旧後の振り返り
- `development-cycle`: Red Flags テーブルで事故の兆候を検出する
